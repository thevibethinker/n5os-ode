## Context: Unsexy Industries Problem

Alex surfaces a pattern: people from non-technical industries (e.g., garbage collection) who have domain-specific ideas but face dual barriers: (1) inability to build software themselves, and (2) lack of developer ecosystems/APIs in their industry. [^3]

**Evidence:**
- "I talk to people who work in the garbage collection who had an idea for an app for 20 years in the garbage collection industry, but I've never been able to build it. And then they also realize, oh, there's no developer ecosystem for garbage, so there's no APIs" [^3]

**Evidence:**

I think the cool thing with Zoho is that because I can set up a brain relatively easily in Zohar, I can have it then have the logic to say, hey, anytime I submit a product, add it to the watch list" [^5]

## Pricing Challenge: Variable Backend Costs

Alex explains Parse's pricing complexity: the same API call can have 100x different costs based on result size, captcha frequency, and anti-bot measures, making usage-based pricing challenging to calculate accurately. [^8]

**Evidence:**
- "It's like causing so many... Part of the issue originally was we kind of have both like usage based pricing, but also the pricing is variable for every website... it's so tricky because if you make an API to like get all the products off of Walmart or something and you want to use. If you put in like a really niche product when you're building, only might only return one page of results... But then if you put in headphones and it returns like 10,000 products, just the network bandwidth is like 100 times more" [^8]

## Strategic Pivot: Usage-Based Pricing

Parse is moving from monthly buckets to usage-based pricing to better align costs and remove friction for one-off use cases. [^9]

**Evidence:**
