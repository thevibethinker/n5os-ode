# After-Action Report: Fixing AAR Generation to Use Semantic Reasoning

**Date:** 2026-01-12
**Type:** build
**Conversation:** con_wCDYWdPur68NGfGX

## Objective

Fix the conversation close system so that After-Action Reports are written by the LLM with semantic understanding, rather than being programmatically generated by Python scripts using regex pattern-matching and template-filling.

## What Happened

V noticed that a recent AAR (for the Agent/CRM cleanup conversation `con_CUmT8IYgH32g6X9K`) contained hollow, template-filled content. Upon investigation, we discovered the root cause: the `conversation_end_full.py` script was *generating and saving* the entire AAR using:

1. `generate_aar_structure()` — which extracted data from SESSION_STATE using string parsing
2. `format_aar_markdown()` — which template-filled a markdown structure by looping through lists
3. `extract_lessons_from_session()` — which used regex patterns like `r"(?:decided|decision|chose)..."` to "extract" decisions

This produced garbage because:
- **Scripts are dumb** — they can match patterns but cannot understand meaning
- **Template-filling produces hollow content** — bullet lists of fragments aren't insight
- **Librarian was reduced to "polishing" garbage** — the semantic analysis role was bypassed

### The Fix

We inverted the architecture:

**Before:**
```
Script → regex extract → template fill → write AAR to disk
Librarian → "enhance" (polish garbage that was already written)
```

**After:**
```
Script → gather raw content → output JSON context bundle (NO AAR)
Librarian → read context → semantically analyze → WRITE real AAR
```

### Key Decisions

1. **Remove all regex-based extraction from scripts** — The `DECISION_PATTERNS` and `OPEN_ITEM_PATTERNS` constants and their associated functions were deleted because pattern-matching cannot produce semantic understanding. The LLM reads the raw content and reasons about what decisions were made.

2. **Scripts output context bundles, not AARs** — The scripts now gather mechanical data (file lists, git status, SESSION_STATE content, debug logs) and output it as JSON for the LLM to process. They do not write any semantic content to disk.

3. **Librarian owns all semantic work** — The Close Conversation prompt now explicitly instructs Librarian to read the context bundle, understand what happened, and write the AAR with real insight. The prompt includes explicit anti-patterns to prevent regression.

## Lessons Learned

### Process

**The "scripts handle mechanics, LLM handles semantics" principle must be enforced rigorously.** It's tempting to add "smart" extraction to scripts — regex patterns, heuristics, template logic — but this always produces garbage for semantic tasks. The correct division is:
- Scripts: file I/O, path manipulation, git status, data gathering
- LLM: understanding, summarization, decision extraction, insight generation

### Technical

**Context bundles are the right interface.** By having scripts output raw content as JSON, the LLM gets full context to reason over. This is better than having scripts pre-process the content, which loses information and introduces errors.
