# Flight Search: V's Personal Travel System

## What It Does

"I say 'flights to LA next weekend' and get results filtered by MY preferences—avoid Spirit, prefer JetBlue, use LaGuardia first."

## The Build Process

**The Need:** Tired of filtering the same airlines out of Google Flights every time

**The Intuition:** "There must be an API for flight data. If I store my preferences locally, I can get exactly what I want."

**The Blocks:**
- Config file for travel preferences (YAML)
- Flight data connection (SerpAPI)  
- Filtering logic (Python script)
- Natural language interface (prompt layer)

**The Build:** Four separate AI conversations, one per component

## Why This Matters

This isn't about flights. It's about recognizing that any repetitive process can become a personalized system.

## Trigger Phrases This Solves

- "I do the same filtering every time"
- "I wish this tool knew my preferences"
- "I'm tired of manual repetitive searches"
- "Can I customize how this works?"

## The Deeper Pattern

1. **Notice repetition** in your workflow
2. **Identify preferences** you apply manually each time
3. **Ask AI:** "How can I automate this with my preferences?"
4. **Build the components** one conversation at a time

## Similar Applications

- Job search with salary/location/role preferences
- Restaurant search with dietary/price/location filters
- Real estate with budget/neighborhood/feature requirements
- Content search with topic/format/length preferences

## The Key Insight

V didn't learn flight APIs. V learned that preferences can be stored in files and applied programmatically.

That insight transfers to every domain.
