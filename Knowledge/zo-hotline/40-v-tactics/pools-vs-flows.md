# Pools vs Flows: Information Architecture

## The Core Principle

Information either flows or it pools. When it pools, it rots.

Every productivity system creates pools: your "Read Later" list, your perfectly organized folders, your comprehensive databases.

These look like storage. They're actually graveyards.

## The Pooling Problem

**Common Pools:**
- "Read Later" bookmark folders
- "Someday/Maybe" task lists  
- Elaborate folder structures
- Comprehensive note systems
- Email archives

Information goes in. It never comes back out when you need it.

## The Flow Alternative

Design for flow, not storage:

```
Input → Triage → Processing → Action/Knowledge → Archive/Delete
```

Information enters, moves through purposeful stages, and exits. Nothing sits. Nothing stagnates.

## Trigger Phrases Callers Use

- "I save things but never find them again"
- "My organization system doesn't work"
- "I have information everywhere but can't use it"
- "My files are organized but useless"

## Flow Design Questions

Instead of "Where should I file this?" ask:
- Where does this need to flow next?
- What action does this enable?
- Who needs to see this and when?
- What happens when it gets there?

## Implementation Examples

**Email Flow:** Inbox → Action Required/FYI/Archive
**Content Flow:** Research → Analysis → Decision → Archive
**Task Flow:** Idea → Planning → Execution → Review

## The Mindset Shift

You're not a librarian organizing books. You're a civil engineer maintaining water systems.

Optimize for flow, not storage.
