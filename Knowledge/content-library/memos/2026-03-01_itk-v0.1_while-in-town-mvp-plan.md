# ITK v0.1 While-In-Town MVP Plan

## One-Line Product
A WhatsApp-first concierge for high-signal travelers moving between NYC and SF, powered by visible founding ITKs (In-The-Know connectors) who surface high-value events and context.

## Strategic Positioning
This is not an events listing app. It is a trust-and-context layer for people whose time is scarce and whose upside comes from being in the right rooms at the right time.

## Locked Decisions (from this conversation)
1. Channel: WhatsApp-first.
2. ITKs: visible members (at least founding ITKs are visible).
3. Rewards: ledger credits now, monetary settlement later.
4. Incentive commitment: founder-committed future payout pool tied to fundraising milestone.
5. v0.1 scope: recommendations only (no intro mediation yet).
6. GTM wedge: While-In-Town travelers, starting NYC and SF.
7. Supply strategy: Parse.bot + manual sourcing + ITK submissions.

## Product Narrative (External)
"When you're in town, we route you to the right events and rooms based on your goals, profile, and what's actually worth your time this week."

## Operating Narrative (Internal)
- Members buy time compression + signal quality.
- ITKs monetize existing behavior (what they already know and attend).
- System moat is compounding profile data and event quality data.

## Technical Decomposition (v0.1)

### 1) Messaging Interface (WhatsApp)
- Intake flow:
  - city + date window
  - goals (meet investors/founders/operators/etc.)
  - vibe constraints (energy, format, neighborhood, timing)
- Mode tags:
  - planned (trip known in advance)
  - opportunistic (last-minute / tonight)

### 2) Core Data Objects
- Traveler Profile
  - identity handle(s), city lanes, preferences, do-not-send constraints, feedback history
- ITK Profile
  - domains, quality score, neighborhoods, event archetypes, reliability score
- Event Card
  - source, datetime, location, audience type, hype level, access level, lead-time risk
- Recommendation Record
  - traveler, ranked options, rationale, sent timestamp, response outcome
- Credit Ledger Entry
  - itk_id, event/recommendation contribution, credit amount, status, notes

### 3) Event Supply Ingestion
- Parse.bot pulls for public sources.
- Manual curation for private/high-signal opportunities.
- ITK submission channel (simple structured format) to capture high-trust supply.

### 4) Ranking + Recommendation Layer
- Prompt-based ranking with human override.
- Output format per recommendation:
  - top 3 options
  - why this fits this person now
  - expected upside
  - confidence + caveat

### 5) Feedback Capture Loop
- Post-recommendation feedback:
  - attended? (yes/no)
  - value score
  - crowd fit score
  - "would go again" signal
- This updates both traveler preferences and ITK/event quality priors.

### 6) Operator Console (Lean)
- Single internal dashboard for:
  - active travelers
  - pending recommendation jobs
  - ITK contribution queue
  - ledger status
- No member-facing app required for v0.1.

### 7) Payments and Credits (v0.1)
- Credits accrue now; payouts deferred.
- Rules visible to founding ITKs upfront.
- Settlement policy tied to fundraising milestone and documented commitment.

## Scope Boundaries (What v0.1 explicitly does NOT include)
1. No intro mediation.
2. No automated payouts.
3. No fully autonomous matching.
4. No broad multi-city expansion.
5. No heavy consumer onboarding funnel.

## Success Metrics for This Test Phase
1. Recommendation alignment score (subjective fit quality).
2. Time saved versus self-search.
3. "Would not have found this otherwise" count.
4. Last-minute opportunity capture rate.
5. ITK contribution consistency (repeat quality submissions).

## 14-Day Action Plan

### Phase A (Days 1-2): Foundation
1. Stand up WhatsApp intake and response workflow.
2. Create data stores for traveler profiles, event cards, recommendations, and credits.
3. Define ITK submission template.

### Phase B (Days 3-5): Supply + First Recommendations
1. Turn on Parse.bot feeds for core sources.
2. Add manual event curation process.
3. Send first recommendations to first traveler(s).

### Phase C (Days 6-9): Feedback + Iteration
1. Capture structured feedback from each recommendation.
2. Tune ranking prompts and event metadata fields.
3. Tighten quality thresholds for what gets recommended.

### Phase D (Days 10-14): ITK Board Activation
1. Onboard a small founding ITK board.
2. Publish visibility + credit rules clearly.
3. Run at least one full recommendation cycle using ITK-submitted events.

## Key Risks and Mitigations
1. Risk: low supply quality.
   - Mitigation: strict curation threshold and ITK quality scoring.
2. Risk: recommendation irrelevance.
   - Mitigation: tighter intake + rapid feedback loop.
3. Risk: ITK distrust about payout.
   - Mitigation: explicit credit ledger transparency and written future payout policy.
4. Risk: operational overload on founder.
   - Mitigation: recommendations-only scope and single-city-lane discipline.

## Next Upgrade Path (v0.2 Candidate)
- Add intro mediation only after recommendation quality is consistently high.
- Introduce controlled automation for response collateral and follow-up orchestration.

## Decision Rule
If v0.1 proves that recommendation quality and repeat usage are high, continue deepening the recommendation + data moat before expanding feature scope.
