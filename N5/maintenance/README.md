# N5 Maintenance System

## Overview

This directory contains **automated maintenance proposals** generated by the n5OS maintenance system. Proposals are created by analyzing issues from the **Squawk List** and **System Upgrades List**, providing actionable solutions for system improvements.

---

## Source Lists

### 1. Squawk List (`Lists/squawk.jsonl`)
**Purpose:** Critical and high-priority maintenance items requiring immediate attention  
**Scope:** Bug fixes, system integrity issues, data loss risks, validation errors  
**Priority Range:** HIGH, MEDIUM, LOW  

### 2. System Upgrades List (`Lists/system-upgrades.jsonl`)
**Purpose:** Planned system improvements, feature enhancements, and workflow optimizations  
**Scope:** New features, performance improvements, UX enhancements, technical debt  
**Priority Range:** H (High), M (Medium), L (Low)  

---

## Proposal Generation

### Scheduling
**Frequency:** Every 3 days  
**Time:** 20:30 ET (after your workday)  
**Model:** Claude Sonnet 4.5  

### Selection Criteria
The auto-proposer selects items based on:

1. **Low Lift, High Reward** — Quick wins with significant impact
2. **Highest Likelihood of Implementation Success** — Clear scope, minimal dependencies
3. **Priority** — Respects the priority field from source lists
4. **Age** — Prefers older unresolved items (FIFO when equal priority)

### Proposal Contents

Each proposal includes:

- **Root Cause Analysis** — Why the issue exists
- **Proposed Solution** — Step-by-step implementation procedure
- **Risk Assessment** — What could go wrong
- **Test Validation Steps** — How to verify success
- **Rollback Procedure** — How to undo if needed
- **Clarifying Questions** — Questions to refine intent, functionality, output, or considerations

### Clarifying Questions

When the proposer needs additional context, it will generate clarifying questions about:

- **Intent:** What is the underlying goal or desired outcome?
- **Functionality:** What specific behavior or features are expected?
- **Output:** What should the end result look like?
- **Considerations:** Are there constraints, dependencies, or edge cases to address?

These questions will be included in the proposal for you to answer before implementation.

---

## Proposal Workflow

1. **Auto-generation** — System creates proposal every 3 days
2. **Review** — You review the proposal and answer any clarifying questions
3. **Approval** — You approve the proposal for implementation
4. **Implementation** — You (or the system) implements the solution
5. **Validation** — Tests are performed to verify success
6. **Closure** — The source item is marked as resolved

## File Structure

```
N5/maintenance/
├── README.md (this file)
└── proposals/
    ├── squawk_squawk-001_YYYY-MM-DD.md
    ├── squawk_squawk-002_YYYY-MM-DD.md
    ├── upgrade_[id]_YYYY-MM-DD.md
    └── ...
```

**Naming Convention:**  
`[source]_[id]_YYYY-MM-DD.md`

- `source` = `squawk` or `upgrade`
- `id` = Original item ID from source list
- `YYYY-MM-DD` = Proposal creation date

## Manual Invocation

To manually generate a proposal:

```bash
python3 N5/scripts/maintenance/squawk_proposer.py [--item-id <id>]
```

Options:
- `--item-id` — Target a specific item ID (skip auto-selection)
- `--source` — Specify source list: `squawk` or `upgrades` (default: auto-select both)
- `--dry-run` — Show what would be selected without generating proposal
