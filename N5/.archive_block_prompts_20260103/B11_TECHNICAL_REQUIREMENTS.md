# B11: Technical Requirements & Specifications

## Objective

Document technical requirements, specifications, architecture decisions, and implementation details discussed in the meeting.

## What to Capture

**Functional Requirements:**
- What the system/solution must do
- User stories or use cases
- Features and capabilities
- Performance requirements

**Technical Specifications:**
- Technology choices
- Architecture decisions
- Integration requirements
- Data models or schemas
- API specifications

**Constraints:**
- Technical limitations
- Compatibility requirements
- Security requirements
- Compliance needs
- Scalability requirements

**Implementation Details:**
- Approach or methodology
- Technical dependencies
- Development sequence
- Testing requirements

**Open Technical Questions:**
- Unresolved technical decisions
- Trade-offs being evaluated
- Research needed

## Output Format

### Functional Requirements

**REQ-001: [Requirement Name]**
- **Description:** [What must be accomplished]
- **Priority:** High/Medium/Low
- **Acceptance Criteria:** [How to know it's done]

### Technical Specifications

**Architecture:**
[High-level technical approach]

**Technology Stack:**
- [Component]: [Technology choice] - [Rationale]

**Integration Points:**
- [System A] ↔ [System B]: [Integration method]

**Data Model:**
[Key entities and relationships]

### Constraints & Requirements

**Security:**
- [Requirement 1]
- [Requirement 2]

**Performance:**
- [Target metrics]

**Compatibility:**
- [Must work with...]

### Open Technical Questions

1. **[Question]**
   - Context: [Why this matters]
   - Options: [Alternatives being considered]
   - Decision needed by: [Date]

## Generation Guidelines

1. Listen for technical language:
   - Requirements, specifications, must-haves
   - Architecture, design, implementation
   - Integration, API, data
   - Performance, scale, security

2. Capture decisions:
   - What was decided?
   - What's the rationale?
   - What were alternatives?

3. Document requirements precisely:
   - Specific, measurable
   - Clear acceptance criteria
   - Priority indicated

4. Note dependencies:
   - What depends on what?
   - Sequence requirements
   - External dependencies

5. Flag unknowns:
   - Unresolved decisions
   - Research needed
   - Risks or concerns

## When to Generate

Generate B11 for meetings involving:
- Technical planning or design
- Architecture decisions
- Integration discussions
- Technical requirements gathering
- Engineering planning

Skip for non-technical business meetings.

## Quality Checklist

Before finalizing, check:
- [ ] Requirements specific and measurable
- [ ] Technical decisions documented with rationale
- [ ] Constraints and limitations captured
- [ ] Dependencies noted
- [ ] Open questions identified
- [ ] Organized logically
- [ ] Useful for engineering team
