# D1.2: Artifact Classifier

## Objective

Create `Skills/build-promote/scripts/artifact_classifier.py` — a CLI tool that classifies every file in a build folder into categories determining where it goes during promotion.

## Context

Build folders contain a mix of operational code, governance docs, runtime state, build scaffolding, and dead artifacts. Before promoting a build to a Skill, we need to know what each file IS so we can route it correctly. This classifier does that analysis.

The Zøde Moltbook promotion (done manually earlier today) established the target skill structure:
```
Skills/<slug>/
├── SKILL.md          # Documentation
├── scripts/          # Operational code
├── assets/           # Governance docs, images, templates
├── prompts/          # System prompts
├── references/       # API docs, specs
└── state/            # Runtime data (new standard)
```

## Scope

### MUST DO
1. Create `Skills/build-promote/scripts/artifact_classifier.py` with these CLI commands:
   - `classify <slug>` — Classify all files in `N5/builds/<slug>/`
   - `classify-cluster <slug>` — Classify across a build cluster (requires lineage_analyzer output)
   - `--dry-run` flag — Preview classification without changes
   - `--json` flag — Machine-readable output
2. Classification categories:

| Category | Pattern Signals | Promotion Target |
|----------|----------------|-----------------|
| `operational_script` | `.py`/`.ts` in `scripts/`, not `__pycache__`, has CLI (`argparse`/`--help`) | `scripts/` |
| `governance` | Persona docs, constitutions, rubrics, policies, `.md` in `artifacts/` with governance keywords | `assets/` |
| `runtime_state` | `.json`/`.jsonl`/`.db` in `workspace/state/`, `workspace/analytics/`, `workspace/memory/`, `workspace/learnings/` | `state/` |
| `prompt` | `.md` in `prompts/` dir or with prompt-related content | `prompts/` |
| `reference` | API docs, specs, `.md` in `references/` | `references/` |
| `static_asset` | Images (`.png`/`.jpg`/`.svg`), templates | `assets/` |
| `build_scaffolding` | `meta.json`, `PLAN.md`, `STATUS.md`, `BUILD.md`, files in `workers/`, `completions/`, `deposits/`, `.n5protected` | Stays in build |
| `obsolete` | `__pycache__/`, `.pyc`, files superseded by newer versions in cluster | Delete |
| `intermediate` | Research notes, exploration artifacts, throwaway drafts | Archive or delete |

3. Use a two-layer classification approach:
   - **Heuristic layer** — File extension, directory position, filename patterns (handles 80% of files)
   - **Content inspection layer** — For ambiguous files, read first 50 lines and look for classification signals (imports, docstrings, frontmatter)
4. For `classify-cluster` mode:
   - Accept a lineage JSON (from lineage_analyzer.py) or re-run analysis
   - Compare same-named files across builds in the cluster
   - Mark files in earlier builds as `obsolete` if a newer build has a newer version
   - Identify the canonical version of each file
5. Output a classification manifest:
```json
{
  "build_slug": "example",
  "total_files": 25,
  "classifications": [
    {
      "path": "N5/builds/example/scripts/webhook.py",
      "category": "operational_script",
      "target": "Skills/<slug>/scripts/webhook.py",
      "confidence": "high",
      "reason": "Python script with argparse CLI"
    }
  ],
  "summary": {
    "operational_script": 5,
    "governance": 3,
    "runtime_state": 8,
    "build_scaffolding": 4,
    "obsolete": 2
  }
}
```

### MUST NOT DO
- Do NOT move, delete, or modify any files
- Do NOT use external packages — stdlib only
- Do NOT use LLM calls (`/zo/ask`) — this is a heuristic classifier
- Do NOT modify meta.json or any build files

### Expected Output
- `Skills/build-promote/scripts/artifact_classifier.py` — Working CLI with `--help`
- Must handle edge cases: empty directories, binary files, symlinks, files without extensions
- Human-readable summary to stderr, machine-readable JSON to stdout

## Technical Notes

- Use `pathlib` for path operations
- Use `argparse` for CLI
- For content inspection, read files as text with error handling for binary files
- The `target` field in classifications should use `<slug>` as a placeholder — the actual skill slug is determined during promotion, not classification
- Confidence levels: `high` (clear pattern match), `medium` (heuristic with some ambiguity), `low` (content-based guess)
