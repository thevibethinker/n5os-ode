# D1.1: Build Lineage Analyzer

## Objective

Create `Skills/build-promote/scripts/lineage_analyzer.py` ‚Äî a CLI tool that scans all `N5/builds/*/meta.json` files and constructs a lineage graph clustering builds by capability.

## Context

There are 90+ build folders in `N5/builds/`. Many are related ‚Äî e.g., `zo-hotline`, `zo-hotline-v4`, `zo-hotline-v5`, `zo-hotline-v6`, `zo-hotline-v7` are all versions of the same capability. Others are one-offs (research, newsletters). The lineage analyzer needs to cluster related builds and identify which are promotable to Skills.

## Scope

### MUST DO
1. Create `Skills/build-promote/` directory with `scripts/` subdirectory
2. Create `Skills/build-promote/scripts/lineage_analyzer.py` with the following CLI commands:
   - `scan` ‚Äî Full analysis of all builds, outputs cluster map
   - `cluster <slug>` ‚Äî Show the cluster a specific build belongs to
   - `promotable` ‚Äî List builds that are candidates for promotion to Skills
3. Scan all `N5/builds/*/meta.json` files. For builds without meta.json, use directory name and file contents as heuristics.
4. Cluster builds using these signals (ranked by reliability):
   - **Explicit lineage** ‚Äî `parent_build` fields in meta.json, Pulse lineage data
   - **Shared deliverable paths** ‚Äî Two builds listing the same output file path
   - **Slug similarity** ‚Äî Version suffixes (`-v2`, `-v3`, `-v4`), shared prefixes
   - **Status progression** ‚Äî `superseded` builds point to successors
5. For each cluster, identify:
   - `canonical_build` ‚Äî The latest/active/most complete build
   - `existing_skill` ‚Äî Check if `Skills/<related-name>/` already exists
   - `promotion_status` ‚Äî One of: `no_skill`, `has_skill_but_may_have_stale_build_artifacts`, `already_promoted`, `not_promotable` (one-off/research)
6. Determine promotability criteria:
   - Status is `complete` or `finalized`
   - Not already promoted (no `transition_note` in meta.json)
   - Has operational artifacts beyond just scaffolding (scripts, state, configs)
   - Is the canonical build in its cluster
7. Output JSON to stdout and optionally save to `Skills/build-promote/state/lineage.json`

### MUST NOT DO
- Do NOT move, delete, or modify any build files
- Do NOT create Skills or modify existing Skills
- Do NOT modify any meta.json files
- Do NOT use external packages ‚Äî stdlib only

### Expected Output
- `Skills/build-promote/scripts/lineage_analyzer.py` ‚Äî Working CLI with `--help`
- Script must handle builds without meta.json gracefully
- Script must work with relative paths from `/home/workspace`

## Technical Notes

- Use `pathlib` for all path operations
- Use `argparse` for CLI
- Use `json` module for meta.json parsing
- For slug similarity, consider: shared prefix (strip `-v\d+$`, `-fix$`, `-debug$`), Levenshtein is overkill ‚Äî simple prefix matching suffices
- For promotability, check if `Skills/*/SKILL.md` exists with a matching name
- Print human-readable summary to stderr, machine-readable JSON to stdout
