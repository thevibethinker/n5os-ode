# D2.1: Promotion Executor

## Objective

Create `Skills/build-promote/scripts/promote.py` — the main promotion script that moves files, adapts paths, generates SKILL.md, and finalizes the build.

## Context

This script does the actual work of transforming a build into a skill. It consumes output from the lineage analyzer (D1.1) and artifact classifier (D1.2), then executes the migration. The Zøde Moltbook promotion established the pattern — this script automates it.

Key principle: **AI handles mechanical adaptation; human retains authority on naming and what to keep.**

## Scope

### MUST DO
1. Create `Skills/build-promote/scripts/promote.py` with these CLI commands:
   - `run <slug> [--name <skill-slug>] [--dry-run] [--yes] [--cluster]`
   - `preview <slug>` — Show what would happen (alias for `run --dry-run`)

2. **Promotion workflow (in order):**

   **Step 1: Pre-flight**
   - Run artifact_classifier.py on the build (or load cached classification)
   - Validate build status is `complete` or `finalized`
   - Check that target `Skills/<slug>/` doesn't already exist (or `--name` specifies a different slug)

   **Step 2: Name proposal**
   - Default: use build slug
   - Clean up: strip version suffixes (`-v2`, `-v3`), fix prefixes, propose professional name
   - In interactive mode: print proposal, wait for confirmation (stdin prompt)
   - With `--name`: use provided name
   - With `--yes`: use default without asking

   **Step 3: Directory creation**
   - Create `Skills/<slug>/` with standard structure:
     ```
     Skills/<slug>/
     ├── scripts/
     ├── assets/
     ├── prompts/
     ├── references/
     └── state/
     ```
   - Only create directories that will have files (don't create empty `prompts/` if no prompts)

   **Step 4: File migration**
   - Copy files to target locations per classification manifest
   - Mapping:
     - `operational_script` → `Skills/<slug>/scripts/`
     - `governance` → `Skills/<slug>/assets/`
     - `runtime_state` → `Skills/<slug>/state/` (preserving subdirectory structure)
     - `prompt` → `Skills/<slug>/prompts/`
     - `reference` → `Skills/<slug>/references/`
     - `static_asset` → `Skills/<slug>/assets/`
     - `build_scaffolding` → stays (not copied)
     - `obsolete` → not copied

   **Step 5: Path adaptation**
   - Scan ALL `.py` and `.ts` files in the new `Skills/<slug>/scripts/` directory
   - Find and replace hardcoded path references:
     - `N5/builds/<build-slug>/workspace/state` → `Skills/<skill-slug>/state`
     - `N5/builds/<build-slug>/workspace/memory` → `Skills/<skill-slug>/state/memory`
     - `N5/builds/<build-slug>/workspace/staging` → `Skills/<skill-slug>/state/staging`
     - `N5/builds/<build-slug>/workspace/analytics` → `Skills/<skill-slug>/state/analytics`
     - `N5/builds/<build-slug>/workspace/learnings` → `Skills/<skill-slug>/state/learnings`
     - `N5/builds/<build-slug>/workspace` → `Skills/<skill-slug>/state`
     - `N5/builds/<build-slug>/artifacts` → `Skills/<skill-slug>/assets`
     - `N5/builds/<build-slug>/scripts` → `Skills/<skill-slug>/scripts`
   - Order matters: most specific paths first to avoid partial matches
   - Also check for `sys.path.insert` hacks referencing the build path — remove them
   - Report all changes made

   **Step 6: SKILL.md generation**
   - Auto-generate a SKILL.md with proper frontmatter:
     ```yaml
     ---
     name: <slug>
     description: <derived from meta.json objective or build title>
     compatibility: Created for Zo Computer
     metadata:
       author: va.zo.computer
       promoted_from: <build-slug>
       promoted_at: <date>
     ---
     ```
   - Generate sections: Directory Structure, How to Invoke (CLI examples for each script), Scripts table, Configuration, Dependencies
   - Mark as draft — V reviews before finalizing

   **Step 7: Protection markers**
   - Add `.n5protected` to `Skills/<slug>/state/` if it exists
   - Copy any existing `.n5protected` from the build workspace

   **Step 8: Build finalization**
   - Update `N5/builds/<build-slug>/meta.json`:
     - `"status": "complete"` (if not already)
     - `"completed_at": "<today>"`
     - `"transition_note": "Promoted to Skills/<skill-slug>/. Build folder retained as historical record."`
   - Remove migrated operational files from build folder (keep scaffolding only)
   - Clean up `__pycache__` in the build

   **Step 9: Print summary**
   - Files moved: N
   - Paths adapted: N scripts, M replacements
   - SKILL.md generated: path
   - Build finalized: slug
   - Next step: `python3 Skills/build-promote/scripts/verify.py check <slug>`

3. **`--dry-run` mode:**
   - Execute Steps 1-2 (pre-flight + name)
   - Print the full migration plan (what would move where)
   - Print path adaptations that would be made
   - Do NOT create directories, copy files, or modify anything
   - Exit with status indicating if promotion is feasible

4. **`--cluster` mode:**
   - Load lineage data to identify all builds in the cluster
   - Classify across the cluster (uses artifact_classifier.py cluster mode)
   - Promote only from the canonical build, but check earlier builds for any files missing from canonical

### MUST NOT DO
- Do NOT use external packages — stdlib only
- Do NOT make any changes in `--dry-run` mode
- Do NOT delete the build folder itself
- Do NOT touch running services, ports, or scheduled agents
- Do NOT hardcode specific build slugs — this must be generic

### Expected Output
- `Skills/build-promote/scripts/promote.py` — Working CLI with `--help`
- Must handle: builds without workspace/ dirs, builds with no scripts, builds that already have a matching skill

## Technical Notes
- Use `shutil.copy2` to preserve timestamps
- Use `pathlib` for all path operations
- For path adaptation, process replacements in order of specificity (longest path first)
- The generated SKILL.md should be functional but marked as draft
- Import and call lineage_analyzer.py and artifact_classifier.py functions directly (they're in the same `scripts/` dir)
