# D6.1: SQLite Backend Migration + Foundation Schema

## Objective

Replace the JSON file backend (`applications.json`) with a SQLite database, and lay the foundational schema for the full FounderMaxxing data layer — applications now, accounts/hotline/sessions later.

Two things built simultaneously to avoid migration friction between phases.

## Context

D6 deployed the intake system with JSON file storage at:
- `Personal/Business/foundermaxxing-applications/applications.json`

This works for a handful of applications but won't survive concurrent writes, has no indexing, and makes future expansion painful. V wants the database foundation built now alongside the migration so the schema is coherent from day one.

### Architecture Decision (from D6 conversation)
- **SQLite** = operational store (all writes, CRUD, the live system)
- **DuckDB** = analytics layer (future, Phase 3 — NOT part of this drop)
- SQLite JSON columns for semi-structured data (hotline transcripts, session notes)

## Part 1: Migration — Applications to SQLite

### Database Location
`Personal/Business/foundermaxxing-applications/foundermaxxing.db`

### Applications Table Schema
```sql
CREATE TABLE applications (
  id TEXT PRIMARY KEY,
  full_name TEXT NOT NULL,
  email TEXT NOT NULL UNIQUE,
  current_title TEXT NOT NULL,
  linkedin TEXT NOT NULL,
  raised_capital INTEGER NOT NULL DEFAULT 0,  -- boolean 0/1
  ai_tools TEXT NOT NULL,                      -- JSON array
  invited_by_v INTEGER NOT NULL DEFAULT 0,     -- boolean 0/1
  vouched_for TEXT,                             -- nullable
  status TEXT NOT NULL DEFAULT 'pending',       -- pending|approved|rejected|waitlisted
  applied_at TEXT NOT NULL,                     -- ISO 8601
  reviewed_at TEXT,                             -- ISO 8601, nullable
  founding_member INTEGER NOT NULL DEFAULT 0,  -- boolean 0/1
  notes TEXT,                                   -- admin notes, nullable
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_applications_status ON applications(status);
CREATE INDEX idx_applications_email ON applications(email);
CREATE INDEX idx_applications_applied_at ON applications(applied_at);
```

### Migration Steps
1. Create the SQLite database with schema
2. If `applications.json` exists, import all existing records
3. Update `/api/foundermaxxing-apply` to write to SQLite instead of JSON
4. Update `/api/foundermaxxing-admin` to read/write from SQLite instead of JSON
5. Verify both routes work with the new backend
6. Keep `applications.json` as a backup (do NOT delete)

### API Routes to Update
Both routes already exist on zo.space — this is a backend-only swap:
- `/api/foundermaxxing-apply` — POST handler: insert into SQLite
- `/api/foundermaxxing-admin` — GET/POST handler: query SQLite for list, update status

The admin dashboard (`/foundermaxxing/admin`) and landing page form (`/foundermaxxing`) are unchanged — they talk to the same API shape.

### Email Draft System
The `/api/foundermaxxing-admin` approval flow uses `/zo/ask` to generate personalized email drafts. This still works the same way — the Opus call reads meeting notes from `Personal/Meetings/` and generates the draft. The draft storage moves from JSON to SQLite:

```sql
CREATE TABLE email_drafts (
  id TEXT PRIMARY KEY,
  application_id TEXT NOT NULL REFERENCES applications(id),
  subject TEXT NOT NULL,
  body TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'draft',  -- draft|sent|discarded
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  sent_at TEXT
);
```

## Part 2: Foundation Schema — Future Tables

Create these tables NOW (empty) so the schema is coherent when hotline/account features land:

```sql
-- Accounts: one per accepted member (created on approval)
CREATE TABLE accounts (
  id TEXT PRIMARY KEY,
  application_id TEXT REFERENCES applications(id),
  full_name TEXT NOT NULL,
  email TEXT NOT NULL UNIQUE,
  linkedin TEXT,
  tier TEXT NOT NULL DEFAULT 'founding',  -- founding|standard|launch
  stripe_customer_id TEXT,
  payment_status TEXT DEFAULT 'pending',  -- pending|active|cancelled|paused
  joined_at TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Hotline sessions: one per call/interaction
CREATE TABLE sessions (
  id TEXT PRIMARY KEY,
  account_id TEXT NOT NULL REFERENCES accounts(id),
  session_type TEXT NOT NULL DEFAULT 'hotline',  -- hotline|chat|meeting
  started_at TEXT NOT NULL,
  ended_at TEXT,
  duration_seconds INTEGER,
  summary TEXT,                    -- LLM-generated summary
  raw_data TEXT,                   -- JSON: transcript, metadata, etc.
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_sessions_account ON sessions(account_id);
CREATE INDEX idx_sessions_started ON sessions(started_at);

-- Interactions: granular events within a session
CREATE TABLE interactions (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL REFERENCES sessions(id),
  account_id TEXT NOT NULL REFERENCES accounts(id),
  interaction_type TEXT NOT NULL,   -- question|answer|action|note|insight
  content TEXT NOT NULL,            -- the actual content
  metadata TEXT,                    -- JSON: sentiment, topics, flags
  occurred_at TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_interactions_session ON interactions(session_id);
CREATE INDEX idx_interactions_account ON interactions(account_id);
```

These tables are empty scaffolding. No API routes needed for them yet — they exist so that when D8 (hotline) or future drops land, the schema is already coherent and migration-free.
