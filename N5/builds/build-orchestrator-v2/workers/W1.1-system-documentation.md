# Worker Brief: System Documentation

**Your Mission:** Create the canonical reference document for the Build Orchestrator v2 system.

**Output:** `Documents/System/Build-Orchestrator-System.md`

---

## Context

The build orchestrator pattern was fundamentally misunderstood. It was being treated as API-based parallelization (`/zo/ask`), when the actual intent is:

- **Human-in-the-loop distributed cognition**
- V manually opens new threads and pastes worker briefs
- Orchestrator is a living conversation V returns to
- Workers report back via close hooks (file writes), not API responses
- State flows through persistent documents, not API calls

## Document Structure

Create a comprehensive system doc covering:

### 1. Overview
- What problem this solves (context window limitation)
- Core philosophy: orchestrator = document, not conversation
- When to use build orchestration

### 2. Folder Structure
```
N5/builds/<slug>/
├── BUILD.md          # Living orchestrator document
├── PLAN.md           # Architect's execution plan
├── STATUS.md         # Progress tracking
├── meta.json         # Machine-readable state
├── workers/          # Worker briefs (paste into new threads)
└── completions/      # Worker completion reports
```

### 3. The Flow
1. V invokes build orchestrator
2. Architect creates plan + worker briefs
3. V opens new thread, pastes worker brief
4. Worker executes, closes (no commits)
5. Close hook writes to completions/
6. V returns to orchestrator, reviews completions
7. Orchestrator adapts remaining briefs if needed
8. Repeat for next wave
9. Final close: orchestrator commits (if code build)

### 4. Worker Brief Format
- Frontmatter with worker_id, title, depends_on
- Self-contained context
- Clear success criteria
- What to report back (learnings, pitfalls, concerns)
- Pre-decided thread title

### 5. Thread Title Convention
Format: `[<build-slug>] W<wave>.<seq>: <task-name>`
Examples:
- `[calendar-v2] W1.1: Database Schema`
- `[calendar-v2] ORCH: Orchestrator`

### 6. Build Types
- `code_build` — Git aware, orchestrator commits
- `content` — Multiple articles/documents
- `research` — Multiple research angles
- `general` — Any task distribution

### 7. Dependency Management
- Waves group workers that can run in parallel
- Dependencies prevent out-of-order execution
- Adaptive briefs: update dependent workers based on completions

### 8. Worker Close Behavior
- NO commits (orchestrator does final commit)
- Writes structured completion to `completions/<worker_id>.json`
- Reports: status, summary, artifacts, learnings, concerns
