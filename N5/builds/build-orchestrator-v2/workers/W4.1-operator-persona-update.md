### v2 Build Orchestration Flow

1. **V requests a build or complex distributed work**

2. **Pre-Screen**
   ```bash
   python3 N5/scripts/build_status.py list --incomplete
   ```
   Check for existing incomplete builds that might overlap.

3. **Initialize Build**
   ```bash
   python3 N5/scripts/init_build.py <slug> --title "Title" --type code_build
   ```

4. **Route to Architect**
   Architect creates PLAN.md with:
   - Task decomposition
   - Worker briefs in `workers/` folder
   - Pre-decided thread titles
   - Wave dependencies

5. **Handoff to V**
   - Present worker briefs to V
   - V opens new threads and pastes briefs
   - Workers are CONVERSATIONS, not API calls

6. **Worker Execution**
   - Each worker executes its brief
   - On close, worker writes to `completions/` (NO commits)
   - Uses pre-decided title from brief's `thread_title` frontmatter

7. **Orchestrator Check-In**
   - V returns to orchestrator conversation
   - Orchestrator reads completions:
     ```bash
     python3 N5/scripts/update_build.py status <slug>
     ```
   - **Updates remaining briefs** with learnings from completed workers
   - Launches next wave

8. **Final Close**
   - All workers complete
   - Orchestrator reviews all completions
   - Commits (if code_build)
   - Closes build:
     ```bash
     python3 N5/scripts/update_build.py close <slug>
     ```

### Key Points to Emphasize

- **Orchestrator = Document**: The build folder IS the orchestrator's brain
- **Workers = Manual Threads**: V opens threads and pastes briefs
- **No API Spawning**: NEVER use /zo/ask to spawn workers
- **Pre-Decided Titles**: Format `[slug] W#.#: Task`
- **Workers Don't Commit**: Only orchestrator commits at final close
- **Adaptive Briefs**: Orchestrator updates remaining briefs with learnings from completed workers

### CRITICAL: The Orchestrator Check-In Loop (Formalized Behavior)

This is NOT optional or ad-hoc. After each wave completes, the orchestrator MUST:

1. **Read all completion reports** from `completions/`
2. **Review `notes_for_orchestrator`** field in each completion — workers are encouraged to speak up about:
   - Gaps they identified
   - Better approaches discovered
   - Warnings for downstream workers
   - Implementation decisions that affect others
3. **Update remaining worker briefs** with relevant learnings before launching next wave
4. **Log the gap** if a systemic issue is identified (add to meta.json `gaps` array)

This adaptive behavior is what makes the orchestrator intelligent rather than just a task dispatcher.

### Worker Feedback Mechanism

Workers should include a `notes_for_orchestrator` field in their completion JSON:
```json
{
  "notes_for_orchestrator": [
    {
      "type": "warning|suggestion|gap|decision",
      "for": "W3.2 or general",
      "message": "The schema I implemented differs from spec because..."
    }
  ]
}
```

Encourage workers to speak up — their ground-level observations are valuable.
