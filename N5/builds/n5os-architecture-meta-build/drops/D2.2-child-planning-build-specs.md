# Drop Brief: Child Planning Build Specs

**Mission:** Define the implementation-ready planning builds that should follow from this meta-build, one per major subsystem or transformation domain.

## Context

This meta-build should lead to several separate planning builds that can be launched in future conversations. Each child build should have a clear scope, objective, dependencies, artifacts, risks, and recommended sequencing.

The child builds should cover the main transformation areas without overlapping excessively.

## Scenarios

S1: Comprehensive subsystem coverage
  Given: The redesign spans multiple major concerns
  When: The build index is written
  Then: It covers the main subsystem planning builds without major gaps
  Verify: LLM: Check that kernel, services, object model, runtime boundaries, cognition, zoffice, apps, and git/artifact hygiene are covered or explicitly deferred

S2: Non-overlapping scopes
  Given: Child builds will be launched independently later
  When: Their scopes are defined
  Then: Each build has a specific objective and limited overlap with siblings
  Verify: LLM: Check that each child build has a distinct mission and dependency profile

S3: Implementation readiness
  Given: These are not merely theme buckets
  When: The build index is reviewed
  Then: Each child build includes concrete outputs, risks, and launch notes
  Verify: LLM: Check that each child build entry includes scope, artifacts, sequencing, and risks

## Requirements

For each child build include:
- slug
- objective
- why it exists
- dependencies
- expected artifacts
- sequencing guidance
- risk level
- whether it should be planning-only or execution-capable initially

## Success Criteria

- [ ] Child builds cover the main redesign domains
- [ ] Scopes are distinct enough to launch separately
- [ ] Entries are implementation-ready

## Anti-Patterns (Avoid)

- Do not define vague or overlapping child builds
- Do not skip sequencing logic
