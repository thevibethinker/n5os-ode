# Drop Brief: Typed Object Model and Registries

**Mission:** Define the typed object model and registry architecture for the redesigned N5 system, centered on commitments/work objects and the registries needed for a clean operating substrate.

**Output:**
- `N5/builds/n5os-architecture-meta-build/artifacts/object-model-and-registries.md`
- `N5/builds/n5os-architecture-meta-build/deposits/D1.3.json`

---

## Context

The redesign is converging on typed work objects rather than a narrow task model. The system needs canonical object types, states, relationships, and registries so that execution, orchestration, routing, and storage are coherent.

This Drop should define:
- commitment arc model
- work object families
- lifecycle states and transitions
- registry types
- which registries are kernel vs service-level
- how current task/action systems map into the future model

## Technical Concepts

| Concept | V's Level | Why It Matters Here |
|---------|-----------|---------------------|
| Domain model | High | This defines the canonical substrate |
| Registry architecture | Medium | Needed to stop implicit state and ad hoc lookup |
| State machines | Medium | Needed for commitments and work lifecycle |

## Scenarios

S1: Commitment arc support
  Given: Work often begins as a commitment and unfolds into multiple actions/builds/followups
  When: The object model is defined
  Then: It represents commitment arcs rather than only atomic tasks
  Verify: LLM: Check that the model includes commitment/objective/work decomposition and lifecycle

S2: Registry clarity
  Given: N5 currently relies on many implicit lookups and ad hoc mappings
  When: The registry design is written
  Then: It defines a small set of canonical registries with clear responsibilities
  Verify: LLM: Check that registry roles are explicit and non-overlapping

S3: Backward mapping
  Given: Existing task/action systems already exist
  When: The model is proposed
  Then: It explains how current systems map into the future design without requiring immediate perfection
  Verify: LLM: Check that the design includes an adaptation path from current task/action components
