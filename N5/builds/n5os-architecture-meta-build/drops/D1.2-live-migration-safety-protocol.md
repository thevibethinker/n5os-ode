# Drop Brief: Live Migration Safety Protocol

**Mission:** Define the safe protocol for adapting the N5 filesystem and architecture while the system remains live and actively used.

**Output:**
- `N5/builds/n5os-architecture-meta-build/artifacts/live-migration-protocol.md`
- `N5/builds/n5os-architecture-meta-build/deposits/D1.2.json`

---

## Context

N5 is a live operating environment. We cannot assume downtime, clean-room migration, or perfect readiness. This Drop defines the migration doctrine for changing architecture and filesystem shape without breaking current usage.

The protocol should establish how to handle:
- source/state/runtime/records separation
- wrappers/adapters and compatibility paths
- versioned moves vs in-place edits
- dual-run patterns
- protected paths
- rollback and observability
- sequencing rules for architectural migration

## Technical Concepts

| Concept | V's Level | Why It Matters Here |
|---------|-----------|---------------------|
| Strangler-fig migration | Medium | Lets old and new coexist safely |
| Dual-write / dual-read | Medium | Useful during registry and object migrations |
| Compatibility layer | Medium | Avoids breaking active callers |

## Scenarios

S1: No-downtime architectural adaptation
  Given: N5 is actively in use while restructuring is planned
  When: The protocol is written
  Then: It defines a path to evolve architecture without assuming downtime or instantaneous file moves
  Verify: LLM: Check that protocol explicitly supports live-system migration and coexistence

S2: Safe path adaptation
  Given: Canonical paths will likely change over time
  When: Filesystem migration guidance is written
  Then: It specifies wrappers/adapters/versioning before physical moves
  Verify: LLM: Check that direct breaking path changes are not the default recommendation

S3: Rollback discipline
  Given: Some migrations will fail or need pausing
  When: The protocol is reviewed
  Then: It contains rollback checkpoints, observability requirements, and stop conditions
  Verify: LLM: Check that rollback and interruption handling are explicit, not implied
