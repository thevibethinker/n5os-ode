# Drop Brief: Current-State Architecture Extraction

**Mission:** Produce a grounded architectural decomposition of N5 as it exists today, identifying kernel-like primitives, reusable services, higher-order frameworks, domain apps, state stores, records, and sprawl zones.

**Output:**
- `N5/builds/n5os-architecture-meta-build/artifacts/architecture-review.md`
- `N5/builds/n5os-architecture-meta-build/deposits/D1.1.json`

---

## Context

This meta-build exists to prepare a comprehensive restructuring of N5 into a cleaner architecture while keeping the live system running. This Drop is responsible for the **current-state map**: what exists now, what is mature, what is central, what is accidental, and how the system is actually functioning on top of Zo.

The work should be evidence-based, not aspirational. Treat domain-heavy systems as exemplars when they reveal reusable infrastructure, but do not confuse them with the substrate itself.

## Scenarios

S1: Grounded architecture map
  Given: The current N5 workspace contains mixed infra, apps, state, and artifacts
  When: The review is completed
  Then: It separates reusable substrate from domain applications and residue using real evidence
  Verify: LLM: Check that each major category is grounded in actual observed files/systems and not just desired future state

S2: Core vs non-core discrimination
  Given: Some heavy domain systems contain reusable patterns
  When: They are discussed in the review
  Then: They are treated as exemplars where appropriate without being mislabeled as core substrate
  Verify: LLM: Check that the review distinguishes platform primitives from domain applications clearly

S3: Sprawl identification
  Given: N5 has source/state/runtime/artifact mixing
  When: The review is written
  Then: It identifies the major sprawl vectors and why they matter operationally
  Verify: LLM: Check that at least 5 specific sprawl/failure modes are named with consequences

## Requirements

### Required sections in `architecture-review.md`
- Executive summary
- What N5 fundamentally is today
- Core infrastructure layers
- Domain exemplars and why they matter
- Current deficiencies
- Maturity assessment by subsystem
- Boundary violations causing sprawl
- Strategic implications for redesign

## Success Criteria

- [ ] Review identifies the real current architectural layers
- [ ] Review differentiates mature vs immature areas
- [ ] Review names specific sprawl patterns
- [ ] Review is implementation-relevant for downstream planning

## Anti-Patterns (Avoid)

- Do not write a future-state proposal here
- Do not flatten everything into “core”
- Do not rely on generic software architecture language detached from N5 reality
