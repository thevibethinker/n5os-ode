# D1.1: Task Registry Schema

## Objective
Create the SQLite database schema for V's task management system. This is the foundational data layer everything else builds on.

## Context
V has ADHD and needs a task system that:
- Tracks tasks as "plans of action" not atomic items
- Organizes by Domain > Project > Task
- Tracks latency (time between due and done)
- Supports priority buckets and LIFO ordering
- Links tasks to source conversations

## Schema Requirements

### Tables

**domains**
- Core life/work areas (e.g., "Careerspan", "Personal", "Zo")
- Permanent, rarely change

**projects**
- Under domains
- Types: ephemeral (one-off), permanent (ongoing), recurring
- Has active/archived status

**tasks**
- The core unit
- Fields needed:
  - id, title, description
  - domain_id, project_id (foreign keys)
  - status: pending, in_progress, blocked, complete, abandoned
  - priority_bucket: strategic, external, urgent, normal
  - source_type: conversation, meeting, manual, email
  - source_id: conversation_id, meeting_id, etc.
  - created_at, due_at, completed_at
  - estimated_minutes, actual_minutes
  - parent_task_id (for subtasks/milestones)
  - plan_json (stores the plan of action, adjustable)

**task_latency** (or computed view)
- Track: due_at vs completed_at delta
- Rolling averages for capacity estimation

**task_events**
- Event log: created, started, blocked, unblocked, completed, abandoned, rescheduled
- Enables latency analysis and pattern detection

### Indexes
- tasks by status, priority_bucket, due_at
- tasks by source_type, source_id
- task_events by task_id, timestamp
