# Drop Brief: Map substrate absorption into exported Zoffice

**Mission:** Audit `zoputer-substrate/` and determine which assets must be absorbed into exported Zoffice now, which remain local factory/reference tooling, and which are deprecated from the deployment story.

**Output:** `N5/builds/zoffice-externalization/artifacts/substrate-absorption-map.md`

---

## Context

V wants the exported office package to include substrate transfer capability, but does not want the older substrate architecture to remain the main conceptual shell. The current `Zoffice/capabilities/zo2zo/` surface may already supersede some substrate pieces. Others may still be useful as patterns or implementation sources. This drop owns the classification and disposition map.

## MUST DO
- Inventory the meaningful assets in `zoputer-substrate/`
- Classify each meaningful asset into one of three buckets: absorb now, keep local/reference only, deprecate from deployment story
- Explain the rationale for each classification
- Highlight any gaps where exported Zoffice still lacks a substrate-derived capability V expects

## MUST NOT DO
- Do not modify `Zoffice/` directly in this drop
- Do not assume all substrate artifacts should be copied over
- Do not leave any major substrate area unclassified

## Scenarios

S1: Every major substrate area gets a disposition
  Given: substrate contains skills, shared tooling, docs, and a partial Zoffice mirror
  When: the absorption map is complete
  Then: each major area lands in exactly one disposition bucket with rationale
  Verify: the artifact contains a table covering skills, shared tooling/docs, and partial Zoffice mirror

S2: Export-package expectations are made visible
  Given: V expects exported Zoffice to carry transfer capability
  When: the map identifies absorb-now items
  Then: it clearly identifies which capabilities/docs/code need to show up inside exported Zoffice for that promise to be true
  Verify: the artifact includes an “absorb now” section tied to current `Zoffice/capabilities/zo2zo/` coverage or gaps

S3: Local-only tooling remains safely outside the package story
  Given: this machine is the factory and can retain standalone substrate tooling
  When: the map classifies local/reference-only assets
  Then: it explains why they stay local and why that does not weaken the exported package
  Verify: the artifact includes a “local/reference only” bucket with explicit rationale

## Expected Output
- A clear three-bucket absorption map
- Concrete recommendations for implementation in D2.1
- Explicit identification of any remaining transfer-capability gaps
