# Drop Brief: Audit prior builds and reconcile canonical deployment story

**Mission:** Read the prior Rice/Zoffice/substrate build artifacts and produce a single architecture reconciliation document that explains what the canonical external deployment story is now.

**Output:** `N5/builds/zoffice-externalization/artifacts/architecture-reconciliation.md`

---

## Context

This build is wrapping up a body of already-completed work. The critical problem is narrative and architectural ambiguity, not lack of raw components. Earlier `consulting-zoffice-stack` and `zoputer-substrate` work described a consultancy/substrate architecture. Later Rice builds (`n5os-rice`, `rice-core`, `rice-capabilities`, `rice-staff`, `rice-integration`) established `Zoffice/` as the actual Layer 1 product. V has explicitly clarified that:
- `Zoffice/` is the canonical external product source
- this Zo should not be mutated/install-targeted as part of this build
- substrate should survive, but as transfer capability integrated into the outward package, not as the primary product shell

## MUST DO
- Read the referenced prior plans and current product surfaces
- Produce a precise reconciliation document with sections for: history, current canonical architecture, deprecated elements, surviving substrate role, and local-vs-export boundary
- Reference concrete builds and file paths
- Make the result legible to a future builder who did not live through the prior iterations

## MUST NOT DO
- Do not edit product code or release scripts
- Do not propose installing anything into va’s live Zo
- Do not leave the relationship between `Zoffice/` and `zoputer-substrate/` ambiguous
