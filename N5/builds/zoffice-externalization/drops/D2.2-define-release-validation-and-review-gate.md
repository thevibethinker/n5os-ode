# Drop Brief: Define release validation and review gate

**Mission:** Create the validation and human-review framework that determines whether the next versioned release folder is ready for V to vet before any transfer occurs.

**Output:** `N5/builds/zoffice-externalization/artifacts/package-readiness-checklist.md` and/or supporting validation artifact updates.

---

## Context

The stop point for this build is not deployment. It is a **review-ready release folder**. That means we need an explicit human gate: what gets checked, what evidence is recorded, and what remains out of scope until V approves transfer.

## MUST DO
- Define the validation checklist for review-readiness
- Tie each checklist item to actual evidence or executable validation where possible
- Clarify what this build does **not** prove (e.g. not actual transfer/install on Zoputer)
- Leave D3.1 with a clear rubric for creating the final review artifact

## MUST NOT DO
- Do not create the release folder yet
- Do not conflate package review with actual deployment success
- Do not rely on generic checklist filler

## scope.files
- `Zoffice/scripts/healthcheck.py`
- `Zoffice/scripts/layer2_smoke_test.py`

- `N5/builds/zoffice-externalization/artifacts/package-readiness-checklist.md`
- `N5/builds/zoffice-externalization/artifacts/release-review.md`

## scope.responsibilities
- Validation rubric design
- Evidence mapping
- Human gate definition
- Clarifying scope boundaries

## scope.must_not_touch
- `Zoffice/releases/**`
- `zoputer-substrate/**`
- unrelated Zoffice code beyond what is necessary to define checks

## Scenarios

S1: Review readiness is evidence-backed
  Given: V wants to vet the package before any transfer
  When: the checklist is complete
  Then: each required check points to concrete evidence or a concrete command/result
  Verify: checklist items reference artifacts, files, or validation commands

S2: Non-goals are explicit
  Given: no transfer/install happens in this build
  When: a future reader interprets the checklist
  Then: they can tell what has and has not been validated
  Verify: checklist includes a section called “Not proven by this build” or equivalent

S3: D3.1 can use the rubric directly
  Given: the final wave creates the release folder and review artifact
  When: D3.1 starts
  Then: it can use this checklist as the basis for release-review output without rethinking the gate
  Verify: the checklist structure maps clearly to a final release-review document

## Expected Output
- Practical review checklist with evidence hooks
- Clear non-goals section
- Direct usability by D3.1
