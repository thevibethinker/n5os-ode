# Drop Brief: Eval harness + cutover validation

**Mission:** Build the evaluation and cutover-validation layer that proves the Google-only canonical memory system is good enough or better on memory quality first, then speed and consistency.

**Output:** Evaluation tooling/docs plus a deposit summarizing measured retrieval quality, speed, and cutover readiness.

---

## Context

This build is not complete just because code compiles. V explicitly cares about memory quality first and speed second. This drop owns the semantics of judging whether the migration is actually an upgrade. It also owns the final evidence narrative for cutover readiness.

## MUST DO

- Add an evaluation harness for retrieval quality and latency
- Test canonical retrieval behavior on meaningful semantic-memory queries and profile-scoped searches
- Validate counts/consistency across resources, blocks, vectors, and ANN state after migration
- Update architecture docs/evidence notes with before/after findings
- Make cutover readiness legible: what improved, what remained equal, and what still needs watching

## MUST NOT DO

- Do not build the provider adapter or migration runner from scratch here
- Do not change the canonical contract unless a blocker forces escalation
- Do not claim success without measurable retrieval and consistency evidence

## Scenarios

S1: Retrieval quality is evaluated on meaningful queries
  Given: V cares about semantic-memory quality first
  When: the eval harness runs
  Then: it measures quality on real representative semantic-memory queries rather than only technical smoke tests
  Verify: inspect query set / outputs in `semantic_memory_eval.py` and run the harness

S2: Speed is measured after quality, not instead of it
  Given: lower latency is useful only if recall/relevance remain acceptable
  When: the eval harness reports results
  Then: quality findings are primary and latency is reported as secondary evidence
  Verify: inspect output ordering/report structure from the eval harness

S3: Cutover readiness includes consistency checks
  Given: a migration can appear successful while leaving broken counts or stale ANN state
  When: cutover validation runs
  Then: it checks resources/blocks/vectors consistency and ANN rebuild/readiness explicitly
  Verify: inspect validation output and baseline doc updates

## Success Criteria

- [ ] Eval harness exists and can run against the migrated canonical substrate
- [ ] Retrieval quality evidence is explicit and primary
- [ ] Speed/latency evidence is reported secondarily
- [ ] Consistency and ANN readiness checks are part of cutover validation
- [ ] Architecture docs reflect the new canonical Google-only system

## Anti-Patterns (Avoid)

- Don’t use only toy or purely mechanical checks
- Don’t optimize for latency while masking relevance regressions
- Don’t declare cutover readiness without count/ANN consistency checks
