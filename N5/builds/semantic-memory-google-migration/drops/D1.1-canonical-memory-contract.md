# Drop Brief: Canonical memory contract on `brain.db`

**Mission:** Make `brain.db` the single canonical active semantic-memory database, remove runtime ambiguity around `vectors_v2.db`, and document the new contract clearly without breaking graph/relationship functionality.

**Output:** Updated canonical-memory code/docs plus a deposit describing the new contract and any migration assumptions.

---

## Context

The current memory system behaves like a split-brain substrate: `brain.db` contains legacy semantic-memory data plus graph/relationship tables, while `n5_memory_client.py` silently prefers `vectors_v2.db` when present. V explicitly approved `brain.db` as the successor DB and wants ambiguity removed. This drop owns the architectural substrate contract, not the Gemini provider implementation itself.

## MUST DO

- Update active-path logic so `brain.db` is the canonical semantic-memory DB
- Remove runtime auto-preference for `vectors_v2.db` in active retrieval paths
- Preserve graph/relationship functionality already stored in `brain.db`
- Clarify the architecture in docs so future instances do not reintroduce dual-active-store confusion
- Keep the code path simple enough that embeddings/indexes are treated as rebuildable derivatives of canonical blocks/resources

## MUST NOT DO

- Do not implement Gemini API calls directly unless a small interface hook is required
- Do not own indexing/reindex pipeline changes beyond what is necessary for the canonical contract
- Do not delete or mutate legacy `vectors_v2.db` data as part of this drop
- Do not tune reranking behavior or evaluation logic
