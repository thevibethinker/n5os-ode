# Drop Brief: Migration tooling + reindex orchestration

**Mission:** Build or adapt deterministic migration tooling that re-embeds canonical resources into `brain.db`, supports safe reruns, and makes the full migration operationally manageable.

**Output:** Migration/reindex tooling updates plus a deposit summarizing resumability, dry-run behavior, and state tracking.

---

## Context

After the canonical contract and Gemini provider are in place, the system needs a reliable way to migrate/rebuild semantic memory into `brain.db`. This drop owns the mechanics of migration and reindex orchestration, not the provider adapter or final evaluation judgment.

## MUST DO

- Create or adapt a migration runner for canonical reindex into `brain.db`
- Preserve or add resumability/state visibility so large reindex jobs are manageable
- Add dry-run support where practical, or clear non-destructive preview behavior if the existing structure makes true dry-run impossible
- Keep state transitions visible and auditable
- Ensure the flow can be rerun without compounding broken or duplicate state

## MUST NOT DO

- Do not alter provider adapter logic beyond required usage wiring
- Do not own evaluation methodology or final cutover judgment
- Do not mutate or delete legacy DBs as part of execution
- Do not expand scope into graph extraction or non-memory subsystems

## Scenarios

S1: Migration can preview scope safely
  Given: a full migration can be expensive and risky
  When: the migration runner is invoked in dry-run or preview mode
  Then: it reports intended scope/progress behavior without destructive cutover
  Verify: run the migration command with dry-run/help and inspect output

S2: Migration can resume after interruption
  Given: large corpus reindex jobs may stop mid-run
  When: the migration runner restarts
  Then: it can resume or safely rerun without duplicating corrupted state
  Verify: inspect resume/state logic and run a small interrupted/restarted test if feasible

S3: State is visible and auditable
  Given: reindexing is a long-running mechanical operation
  When: the runner executes
  Then: progress/state files or logs make current state visible enough for troubleshooting
  Verify: inspect updated state files/logging paths and run a sample migration step

## Success Criteria

- [ ] A canonical migration/reindex entrypoint exists
