# D2.1: Zo.space Follow-Up Page

## Context

You are building the personalized follow-up page system for the Vibe Thinker Hotline. After a caller has a substantive conversation with Zoseph, they receive an SMS with a link to a personalized zo.space page. This page contains their conversation summary, ready-to-paste prompts, getting-started steps, and relevant links.

**This page IS the product demo.** A caller who just learned about Zo Computer sees a page that Zo built for them in real-time. The meta-demonstration value is critical.

**V's zo.space is at:** `va.zo.space`

## Scope

### Routes You Own
- zo.space API route: `/api/hotline/create-page` — CREATE — Accepts call data, generates page content, creates the page route
- zo.space page route: `/hotline/:slug` — CREATE — The follow-up page template

### Must Not Touch
- `Skills/zo-hotline/scripts/hotline-webhook.ts` — D2.2 will integrate
- `Skills/zo-hotline/prompts/zoseph-system-prompt.md` — D1.2 / D2.2 own this
- Any existing zo.space routes

## MUST DO

### 1. Create the Follow-Up Page Template

Create a zo.space page route at `/hotline/demo` that serves as the template design. This is a PUBLIC page.

**Design requirements:**
- Mobile-first (most callers will open this from their phone after receiving the SMS)
- Clean, minimal, warm — not corporate or flashy
- Dark theme (zinc-900 background) to match Zo aesthetic
- Good typography hierarchy
- The page should feel personal, not templated

**Sections:**
1. **Header:** "Your Vibe Thinker Hotline Follow-Up" + caller's first name if available + date
2. **What We Talked About:** 2-3 sentence summary of the conversation
3. **Try This on Zo:** Ready-to-paste prompt(s) in a code block with a copy-to-clipboard button. This is the most important section — it gives the caller a concrete first action.
4. **Your First 15 Minutes:** Step-by-step guide (3-5 steps) tailored to their use case and technical level. Example: "1. Log in to your Zo at [handle].zo.computer → 2. Paste the prompt above into the chat → 3. Watch Zo set it up → 4. Check back tomorrow morning for results"
5. **Keep Going:** Links to Discord (discord.gg/zocomputer), support (support.zocomputer.com), and 1-2 relevant community examples if applicable
6. **Footer:** "This page was built by Zo Computer after your call — in real-time. That's the kind of thing you can build too." + link to zocomputer.com

**Interactive elements:**
- Copy button on prompt blocks (using navigator.clipboard API)
- Smooth scroll between sections
- Optional: subtle animation on load

### 2. Create the Page Generation API Route

Create a zo.space API route at `/api/hotline/create-page` that:

**Accepts POST with JSON body:**
```typescript
{
  callId: string,        // VAPI call ID
  callerName?: string,   // First name if collected
  summary: string,       // Call summary
  pathway: string,       // explorer | builder | comparison
  level?: number,        // Technical level 1-5
  primaryInterest: string, // Main topic discussed
  prompts: string[],     // Ready-to-paste prompts for Zo
  nextSteps: string[],   // Step-by-step instructions
  communityExamples?: string[] // Relevant examples to show
}
```

**Does:**
1. Generates a URL-safe slug from callerName + primaryInterest + date (e.g., `amanda-climate-policy-feb18`)
2. Creates a PUBLIC page route at `/hotline/<slug>` using `update_space_route`
3. The page route embeds the data directly in the component (no API call needed at render time)
4. Returns: `{ url: "https://va.zo.space/hotline/<slug>", slug: "<slug>" }`

**Important implementation detail:** The API route must use the Zo API to create the page route. Since zo.space API routes can't call `update_space_route` directly (it's a Zo tool, not a JS function), the API route should use `fetch` to call the Zo API:

```typescript
const response = await fetch("https://api.zo.computer/zo/ask", {
  method: "POST",
  headers: {
    "Authorization": `Bearer ${process.env.ZO_API_KEY}`,
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    input: `Create a public zo.space page route at /hotline/${slug} with this content: ...`
  }),
});
```

Wait — this is complex and fragile. **Simpler approach:** The API route generates the full React component source code as a string, then calls the Zo API to create the route. The page content is baked into the component at creation time.

**Even simpler approach (RECOMMENDED):** Don't use an API route at all. D2.2's webhook code will call `/zo/ask` directly to create the page. This drop should just create:
1. A **template page** at `/hotline/demo` showing the design with sample data
2. A **helper function** (as a code snippet in the deposit) that generates the React component source for any given call data

This keeps the architecture simple — no API-to-API chains.
