# W2.1 — Sync/backfill correctness + repair tooling

## Goal
Make `sync`/`backfill` achieve **file↔DB consistency** and provide repair tooling (idempotent, safe, and merge-aware).

## Wave-1 Learnings (must incorporate)
- **Calendly sync now writes DB records** (18 links) via `Integrations/calendly/sync_links.py` (worker W1.4).
- **Schema drift:** `N5/scripts/content_library.py` still assumes legacy `type` (`link|snippet`) while the real DB uses `content_type` (article/link/snippet/social-post/...). Any calls to `lib.add()` can break.
- **Preferred metadata pattern:** Use the existing normalized `tags` table for flexible context (audience/priority/use_when/etc.), not one-off columns, unless strongly justified.
- **Merge policy:** Never overwrite curated context (only fill missing).

## Context
- Backfill today: `N5/scripts/content_backfill.py` scans filesystem and calls `ingest_file(move=False)`.
- We previously observed **orphaned Calendly files** that were not in DB.
- W1.2 proposed schema additions (at minimum): `subtype`, `summary`, `managed_fields`.

## Tasks
1. **Root-cause the historical miss** (Calendly files not in DB previously):
   - Was it because Calendly files lived outside scan paths?
   - Was it record_exists keyed on source_file_path mismatch?
   - Was it because the files were never ingested at all (sync_links wrote files but no ingest)?
   - Document exactly why.

2. **Design a canonical `sync` model** (fixed point):
   - `python3 N5/scripts/content_library.py sync` should reconcile:
     - files present → ensure DB record exists
     - DB record exists but file missing → flag (no destructive auto-delete)
     - mismatched `source_file_path` → propose repair
   - Second run should be 0 changes (**idempotence**).

3. **Schema alignment tasks (no commits, but can propose changes):**
   - Ensure `content_library.py` API aligns with the real schema:
     - read/write `content_type` (not `type`) OR provide robust alias layer
     - add optional `subtype`, `summary`, `managed_fields`
   - Decide where `managed_fields` is tracked (DB column vs frontmatter vs both). If both, specify precedence.

4. **Repair tooling (proposal + minimal implementation plan):**
   - orphan file → create DB record (merge-aware)
   - orphan DB record → flag with recommended action
   - allow `--dry-run`
   - produce a structured report (json)

5. **Tests to define**
   - Sync idempotence (run twice)
   - Orphan file ingestion
   - Curated context preservation (tags/frontmatter)
