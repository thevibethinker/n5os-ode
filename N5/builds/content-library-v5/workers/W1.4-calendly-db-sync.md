# W1.4 — Calendly sync → DB (non-destructive enrichment)

## Goal
Make Calendly sync produce:
1) the existing high-quality markdown files AND
2) corresponding DB records, with contextual fields that accumulate safely.

## Context
- Current: `Integrations/calendly/sync_links.py` writes files under `Knowledge/content-library/links/calendly/`.
- Current: those files are *not* being indexed into DB (or sync/backfill is not picking them up).
- Requirement: add “context around when to use” (audience/priority/use_when), but don’t overwrite curated context.

## Tasks
1. Audit current Calendly sync behavior.
2. Propose strategy to write DB records:
   - Option A: call `content_ingest.py` on each generated file
   - Option B: call `content_library.py add` directly
   - Option C: embed a dedicated DB upsert in sync_links.py
   Choose and justify.
3. Define stable identifiers:
   - item id/slug derived from Calendly event type slug or UUID
4. Implement merge policy:
   - if DB already has `audience/priority/use_when`, do not overwrite
   - if missing, write defaults (or leave blank)
5. Propose minimal “usage context” defaulting:
   - e.g., infer from event type name (“friends and family”) but keep conservative
