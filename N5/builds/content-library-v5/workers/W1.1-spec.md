# W1.1 — Spec (taxonomy + frontmatter + merge semantics)

## Goal
Produce a concrete, implementable spec for Content Library v5 covering:
1) taxonomy (types + subtypes), 2) canonical frontmatter fields per type, 3) merge semantics (defaults vs curated context), and 4) invariants for file↔DB sync.

## Context
- Canonical storage: `Knowledge/content-library/`
- DB: `N5/data/content_library.db`
- Current scripts:
  - `N5/scripts/content_ingest.py`
  - `N5/scripts/content_backfill.py`
  - `N5/scripts/content_library.py`
- Calendly “good file” example: `Knowledge/content-library/links/calendly/*.md`

## Non-negotiables (from V)
- Normalize to **clean original text** + **short summary in frontmatter**.
- No raw web scrape / UI garbage.
- Calendly: must be in DB and include usage context.
- Context policy: write defaults, **never overwrite existing curated context**.
- Content Library is a first port of call for assets; not the routing brain.

## Deliverables
1. A spec section (bulletproof) answering:
   - **Type vs subtype:** which concepts are `type` vs `subtype`.
   - **Frontmatter schema per type** (required/optional fields).
   - **Context fields** (e.g. `audience`, `priority`, `use_when`) and merge rules.
   - **Stable ID strategy** (recommended) and collision handling.
2. A recommended directory layout confirmation (what belongs where).
3. Test cases list (golden inputs/outputs).

## Guidance / Starting Point
Recommend keeping existing top-level types (article/link/snippet/etc.) and using `subtype` for:
- `link` subtypes: `scheduling-link`, `profile`, `resource`, `reference`, `download`, `internal-tool`, etc.
- `article` subtypes: `blog`, `newsletter`, `essay`, etc.

### Merge Semantics (must specify)
Define field categories:
- **managed (automation-owned):** e.g. `url`, `source`, `source_url`, `source_file_path`, `word_count`, timestamps
- **curated (human/context):** e.g. `audience`, `priority`, `use_when`, notes

Rule: automation may only **fill missing** curated fields; never overwrite non-empty.
