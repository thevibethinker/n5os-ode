# W2.2 — Migration/dedupe for existing library items

## Goal
Design a **safe migration + dedupe plan** for existing Content Library items, without data loss:
- dedupe where needed
- retroactively add subtype/context
- handle legacy Calendly IDs vs new deterministic scheme

## Wave-1 Learnings (must incorporate)
- **Legacy Calendly duplicates exist:** ~7 `vrijen_*` scheduling items in DB from `json_migration` coexist with new `calendly_*` IDs created by the updated `Integrations/calendly/sync_links.py` (W1.4).
- **Stable ID strategy is now a requirement** (W1.1): deterministic IDs, collision suffixing, immutable once created.
- **Curated context must not be overwritten** (audience/priority/use_when/context/purpose).

## Context
- DB has many legacy items with `source='json_migration'` and often no `source_file_path`.
- Files include:
  - high-quality structured Calendly link files (in `Knowledge/content-library/links/calendly/`)
  - mixed-quality older files

## Tasks
1. **Dedupe strategy proposal (with risk scoring):**
   - duplicates by URL (strong signal)
   - duplicates by (content_type, subtype, normalized_title)
   - duplicates by external_id (Calendly UUID) if we store it
   - collisions from slug generation

2. **Calendly ID migration plan (specific):**
   - Decide target canonical IDs (likely `calendly_*` deterministic IDs)
   - Create mapping `old_id -> new_id` (redirect table or JSON log)
   - Decide what to do with callers that reference old IDs
   - Preserve curated tags from legacy records if they exist

3. **Subtype retrofit plan:**
   - Mark Calendly links as `content_type=link`, `subtype=scheduling-link`
   - Propose subtypes for other common link buckets (profile/resource/reference) without expanding top-level types.

4. **Clean-up plan for known-bad raw scrape files (if any remain):**
   - convert to structured file format
   - keep clean original text (not UI chrome)
   - add short summary in frontmatter
