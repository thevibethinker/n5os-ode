# W1.2 — DB schema + migrations + API alignment

## Goal
Design and (optionally) implement DB schema changes + migration approach so that:
- DB supports `subtype` and curated usage context fields.
- `content_library.py` and `content_ingest.py` converge on one schema model.
- Calendly and other file-based assets can be indexed reliably.

## Context
- DB: `N5/data/content_library.db`
- Current schema appears richer than `content_library.py`’s legacy dataclass assumptions.
- `content_ingest.py` already writes `content_type`, `source_file_path`, etc.
- `content_library.py` has legacy `type` constraint ('link','snippet') but also refers to content types.

## Tasks
1. Inspect current DB schema (`sqlite3 ... .schema`) and identify mismatches with `content_library.py`.
2. Propose canonical schema v5:
   - Add columns: `subtype`, and curated context columns (or JSON) for scheduling links:
     - `audience` (JSON array or normalized tags?)
     - `priority` (enum-ish)
     - `use_when` (text)
   - Decide whether to store curated context in dedicated columns vs `tags` table vs `notes` JSON.
3. Propose migration steps (non-destructive):
   - `ALTER TABLE` additions
   - backfill defaults
   - add indexes
   - constraints / uniqueness strategy
4. Propose API alignment strategy:
   - Update `content_library.py` dataclass to reflect real schema
   - Ensure backward compatibility for callers

## Guardrails
- Do NOT delete data.
- Avoid a “hard reset” migration.
- If implementing migrations, prefer idempotent SQL (check if column exists).
