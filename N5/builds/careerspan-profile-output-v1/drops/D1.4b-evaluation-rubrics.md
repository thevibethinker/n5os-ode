# D1.4b: Evaluation Rubrics

## Objective

Transform panel purposes into operational rubrics that can be applied to candidate data systematically, producing auditable output with reasoning traces.

---

# Rubric: Panel 1 — The Thesis

## Required Elements

| Element | Description | Must Include |
|---------|-------------|--------------|
| **Candidate type/mold** | Who is this person professionally? | One clear descriptor (e.g., "pragmatic shipper", "researcher-practitioner") |
| **Value proposition** | What do you get if you hire them? | Specific capability or combination |
| **Trade-off** | What are they NOT good at? | Honest limitation — no perfect candidates |
| **Evidence anchor** | What backs up the claims? | At least one citation |

## Quality Criteria

| Criterion | Pass/Fail Check |
|-----------|-----------------|
| **Single paragraph** | 4-5 sentences maximum. Fail if >5. |
| **Buried lead** | First sentence conveys core verdict. Fail if takes >10 seconds to understand who this is. |
| **Specificity** | Uses actual evidence, not generic praise. Fail if "highly motivated", "team player" without examples. |
| **Trade-off named** | Must identify a limitation. Fail if no trade-off mentioned. |
| **Standalone value** | Reader can make preliminary decision from this alone. |

## Evidence Requirements

Each claim must have:
- **Source citation** — Where was this found? `[Evidence: {source}]` e.g., `[Evidence: LinkedIn post about RAG deployment]`
- **Specific example** — Not "great engineer" but "built and scaled RAG system serving 100K queries"
- **Verifiability** — Could you point a skeptic to the evidence?

## Length Constraints

| Constraint | Limit | Enforcement |
|------------|-------|-------------|
| **Total sentences** | 4-5 sentences maximum | Count sentences, fail if >5 |
| **Sentence length** | No run-on paragraphs | Break if >35 words per sentence |
| **Word count** | 75-125 words | Fail if <50 or >150 words |

## Auditing Output

The thesis must include a reasoning trace at the bottom:

```markdown
### Thesis Reasoning
- **Candidate type identified from:** [source]
- **Value proposition derived from:** [source]
- **Trade-off identified from:** [source]
- **Evidence sources used:** [list of sources cited]
```

# Rubric: Panel 2 — Hard Skills (Bar Cleared)

## JD Requirements to Check

### FlowFuse AI Engineer — Required Skills

| JD Requirement | Operational Check | Evidence Sufficient If |
|----------------|-------------------|------------------------|
| **Strong generalist engineering skills (Node.js)** | Has Node.js experience building production systems | Project using Node.js shipped to users |
| **Hands-on experience building with LLMs in production** | Has deployed an LLM feature beyond prototype | LLM feature live with users or in production environment |
| **Familiarity with vector databases, embeddings, retrieval systems** | Has used vector DB (Pinecone, Weaviate, pgvector, etc.) | Code or project using embeddings + retrieval |
| **Comfort with ambiguity and bias toward shipping incrementally** | Evidence of iterating, shipping partial solutions | Shipped MVP, iterated based on feedback |
| **Excellent written communication** | Has documented patterns, decisions, or technical content | Blog post, README, docs contribution, design doc |
| **Bonus: Node-RED or low-code platform experience** | Has used Node-RED or similar low-code tools | Project on Node-RED or similar platform |

## Evidence Sufficiency Levels

| Level | Definition | Evidence Examples |
|-------|------------|-------------------|
| **Strong** | Clear, direct evidence from production work | "Built RAG system for Y using OpenAI + pgvector, now serving 100K daily queries" |
| **Moderate** | Indirect evidence or pre-production | "Prototyped LLM feature using LangChain, in internal testing" |
| **Weak** | Tangential or educational-only | "Completed LLM course, built demo with OpenAI API" |
| **Not Found** | No evidence found or irrelevant | Mentioned AI in general, no specific work |

## Verdict Determination

| Verdict | Condition | What to Output |
|---------|-----------|----------------|
| **Cleared** | All required items (first 5) have Strong or Moderate evidence | `Cleared: [requirement] — [1-sentence proof point]` |
| **Not Cleared** | Any required item has Weak or Not Found evidence | `Not Cleared: [requirement] — [gap reason]` |
| **Unclear** | Evidence exists but is ambiguous or outdated | `Unclear: [requirement] — [ambiguity note]` |

**Overall Bar Verdict:**
- **Bar Cleared:** All required items have Strong/Moderate evidence
- **Bar Not Cleared:** ≥1 required item has Weak/Not Found evidence
- **Bar Unclear:** Majority required items are Unclear

## Handling Ambiguous Signals

| Situation | Resolution |
|-----------|------------|
| **Old experience** (3+ years ago) | Flag as "stale evidence," downgrade to Moderate |
| **No tech stack match** (e.g., Python instead of Node.js) | Still assess engineering capability, note as "language transfer potential" |
| **Prototype only** | Downgrade to Moderate unless accompanied by shipping track record |
| **Company claims without details** | Mark as Unclear, request more evidence |

## What "Bar Cleared" Means Operationally

For FlowFuse AI Engineer, "bar cleared" means:
- Can ship production AI features (not just prototypes)
- Can work in Node.js environment (or transfer easily)
- Can build retrieval systems (vector DB familiarity)
- Can communicate in writing (docs, decisions)
- Comfortable with uncertainty (evidence of incremental shipping)

If any required area is missing, the candidate does NOT clear the bar for this role — regardless of how impressive their other skills are.

## Auditing Output

Panel 2 must include:

```markdown
### Hard Skills Reasoning
- **Overall verdict:** [Cleared/Not Cleared/Unclear]
- **Not Cleared items:** [list if any]
- **Weaknesses requiring explanation:** [list]
- **Bonus items found:** [list from JD bonuses]
- **Evidence sources:** [list of sources used for each requirement]
```

# Rubric: Panel 3 — Differentiators

## What Qualifies as a Differentiator

**Differentiators are NOT table stakes.** Table stakes are what every qualified candidate should have. Differentiators are rare, unusual, or remarkably strong.

### Table Stakes (DO NOT include)
- "Good communicator"
- "Team player"
- "Fast learner"
- "Hard worker"
- "Problem solver"
- "Detail-oriented"
- "Proactive"

### Actual Differentiators (DO include)
- **Technical rarity:** "Built production RAG at scale," "Fine-tuned models for specific domains"
- **Unusual combination:** "Product manager turned AI engineer," "AI safety researcher who ships"
- **Domain expertise:** "Built for manufacturing/supply chain," "Experience in regulated industries"
- **Achievement signal:** "Open source maintainer," "Published research," "Built used-by-thousands tool"
- **Culture resonance:** "Emojis in technical writing," "Experienced remote-first builder," "Community builder"

## Differentiator Categories

| Category | What It Is | Examples |
|----------|------------|----------|
| **Technical** | Rare skill or depth | "Fine-tuning experience," "MLOps at scale," "Edge deployment" |
| **Soft/Culture** | Unusual personality or team fit | "Writes with personality," "Humor in technical communication," "Empathy-driven engineer" |
| **Domain** | Industry-specific knowledge | "Manufacturing background," "Healthcare ML experience," "IoT systems" |
| **Achievement** | Proof of exceptional output | "Maintained popular open source project," "Shipped AI feature used by 100K+" |
| **Origin Story** | Unusual path to engineering | "Self-taught bootstrapper," "PhD who shipped product," "Founder-to-engineer" |

## Evidence Requirements

Each differentiator must have:
- **Source citation** — Where was this found? (LinkedIn post, project, bio, etc.)
- **Specific example** — What did they actually do? Not "great leadership" but "led 5-person team through product launch"
- **Verifiability** — Could you point a skeptic to evidence?

**Evidence format:**

```markdown
### Differentiator: [Name]
**Category:** [Technical/Soft/Domain/Achievement/Origin Story]
**Proof point:** [Specific example with source]
```

## How Many to Highlight

| Quantity | When to Use | Guidance |
|----------|-------------|----------|
| **0** | Nothing genuinely differentiates | Explicitly state: "No standout differentiators found" |
| **1** | One truly exceptional trait | Focus on it, don't invent more |
| **2-4** | Sweet spot | Quality over quantity |
| **5+** | Diluted meaning | Pick the top 2-3, discard the rest |

## Edge Cases

| Situation | Resolution |
|-----------|------------|
| **Strong table stakes** | If someone is EXCEPTIONALLY good at communication (e.g., wrote a book on it), it becomes a differentiator — otherwise, skip |
| **Unrelated cool fact** | If it doesn't matter for THIS role (e.g., fluent in Esperanto for AI engineer role), skip |
| **Multiple weak differentiators** | Better to list 1 strong than 3 weak |
| **No differentiators** | Be honest — "Qualified on hard skills, but no standout differentiators" |

## Auditing Output

Panel 3 must include:

```markdown
### Differentiators Reasoning
- **Count:** [number of differentiators]
- **Categories:** [list of categories represented]
- **Rejected candidates for differentiator status:** [if any items were considered but rejected, explain why]
- **Evidence sources:** [list of sources for each differentiator]
```

# Rubric: Panel 4 — Why This Meeting Matters

## Culture Fit Assessment (FlowFuse-Specific)

### FlowFuse Culture Signals (from JD research)

| Signal | What It Means | How to Assess |
|--------|---------------|---------------|
| **Remote-first, async** | Comfort with Slack-based async communication, no need for constant syncs | Evidence of remote work, async collaboration, written communication preference |
| **Informal, personality-forward** | Team adds custom emojis, casual communication style | Evidence of humor, personality in writing, non-corporate tone |
| **Shipping incrementally** | Bias for action over perfection, comfortable with imperfect first versions | Evidence of shipped MVPs, iterative development, "good enough" shipping |
| **Ownership** | Takes responsibility, drives work independently | Evidence of leading features, owning outcomes, autonomy |
| **Clear communication** | Writes well, documents decisions | Evidence of docs, design docs, clear writing samples |

### Culture Fit Checklist

| Item | Question | Evidence Source | Fit/No Fit/Unclear |
|------|----------|-----------------|-------------------|
| **C1** | Do they show evidence of remote work comfort? | LinkedIn, portfolio, bio | |
| **C2** | Do they communicate with personality/informal tone? | LinkedIn posts, GitHub README, writing samples | |
| **C3** | Do they show shipping bias (shipping over perfection)? | Project history, blog posts | |
| **C4** | Do they show ownership/autonomy? | Project descriptions, achievements | |
| **C5** | Is their written communication clear and effective? | Any writing samples available | |

**Culture Fit Verdict:**
- **Strong Fit:** ≥4 items have clear evidence
- **Moderate Fit:** 2-3 items have evidence
- **Uncertain:** 0-1 items have evidence, or mixed signals
- **Poor Fit:** Clear misalignment (e.g., corporate-only communication style, no remote experience)

## Generating Specific Interview Questions

For each exploration area, generate a question that:

1. **Probes directly** — Don't ask "tell me about yourself" (too generic)
2. **Connects to evidence** — References something found in their profile
3. **Ties to FlowFuse** — Makes the connection to the company/role

### Question Templates

| Exploration Area | Template | Example |
|------------------|----------|---------|
| **Shipping bias** | "You [shipped X feature] quickly. How do you balance speed with quality when building AI features?" | "You shipped your RAG prototype in 2 weeks. How do you decide when something is 'good enough' to release?" |
| **Technical depth** | "I see you [technical detail]. How would you apply that to [FlowFuse challenge]?" | "You've worked with vector databases at scale. How would you design retrieval for FlowFuse's workflow automation use case?" |
| **Culture fit** | "Your [personality signal] stands out. How do you [culture-related behavior]?" | "Your GitHub READMEs have a lot of personality. How do you balance being yourself with professional documentation?" |
| **Ownership** | "You [owned X outcome]. How do you approach [ownership scenario]?" | "You led the AI feature launch at [company]. How do you handle ambiguous requirements when owning a feature?" |
| **Origin story** | "You transitioned from [previous role] to engineering. How does that [perspective] help you?" | "You moved from product management to AI engineering. How does that product lens shape how you ship features?" |

### Number of Questions

| Target | Constraint |
|--------|------------|
| **2-3 exploration areas** | Maximum for 30-minute meeting |
| **1-2 questions per area** | Don't overstuff |
| **Total: 2-4 questions** | Ideal range |

## Structuring the "Worth Meeting" Argument

The "worth meeting" section must answer:

1. **What's at stake?** — Why does this conversation matter for FlowFuse?
2. **What will we learn?** — What unknowns will this meeting resolve?
3. **What's the downside of missing it?** — What do we risk by NOT meeting them?

### Structure Template

```markdown
### Why This Meeting Matters for FlowFuse

**Stake:** [What's at stake — why this hire matters]

**Exploration Areas:**
1. [Area 1] — [Question to ask]
2. [Area 2] — [Question to ask]
3. [Area 3] — [Question to ask]

**What We'll Learn:**
- [Specific unknown the meeting resolves]
- [Specific unknown the meeting resolves]

**Downside of Missing:**
- [What we lose by not meeting them]
```

### Handling Thin Culture Signals

| Situation | Resolution |
|-----------|------------|
| **No culture signals found** | State explicitly: "Limited culture signals available. Recommend exploring directly in conversation." |
| **Mixed signals** | Note the mix and suggest probing: "Signals are mixed — shows [positive] but [concern]. Recommend exploring [specific topic]." |
| **Strong technical but weak culture** | Be honest: "Bar cleared on hard skills, culture fit uncertain. Meeting worthwhile to assess alignment." |
| **Strong culture but weak technical** | Be honest: "Culture signals strong, but hard skills not fully cleared. Meeting only if you're willing to consider training." |

## Auditing Output

Panel 4 must include:

```markdown
### Why This Meeting Matters Reasoning
- **Culture fit assessment:** [Strong/Moderate/Uncertain/Poor]
- **Culture signals found:** [list of signals with sources]
- **Missing culture signals:** [list of JD signals not found]
- **Exploration areas rationale:** [why these 2-3 areas were chosen]
- **Question sources:** [which profile evidence inspired each question]
```

# Cross-Panel Auditing

## Consistency Checks

When a candidate profile is evaluated, these cross-panel checks must be performed:

| Check | Description | Failure Mode |
|-------|-------------|--------------|
| **Thesis vs. Hard Skills** | Does the thesis accurately reflect the hard skills verdict? | Thesis says "strong engineer" but Hard Skills says "Not Cleared" |
| **Thesis vs. Differentiators** | Does the thesis mention or align with differentiators? | Thesis claims unique value but no differentiators found |
| **Differentiators vs. Culture** | Are cultural differentiators reflected in Panel 4? | Panel 3 calls out "informal communicator" but Panel 4 doesn't explore it |
| **Hard Skills vs. Meeting Worth** | Does Panel 4 acknowledge the hard skills verdict? | Panel 2 says "Not Cleared" but Panel 4 doesn't mention this risk |

## Overall Output Structure

When D2.1-D2.4 apply these rubrics, the final profile output should be:

```markdown
# [Candidate Name] — Profile

## Panel 1: The Thesis
[Thesis paragraph]

### Thesis Reasoning
[Reasoning trace]

|-------------|---------|-------------|
| [JD req 1] | [Cleared/Not Cleared/Unclear] | [Evidence] |
| [JD req 2] | [Cleared/Not Cleared/Unclear] | [Evidence] |
| ... | ... | ... |

**Overall Bar:** [Cleared/Not Cleared/Unclear]

### Hard Skills Reasoning
[Reasoning trace]
```
