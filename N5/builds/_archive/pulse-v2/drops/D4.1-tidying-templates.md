# D4.1: Tidying Drop Templates

## Objective

Create the 5 hygiene Drop templates that run after BUILD completes.

## Deliverables

Create `Skills/pulse/drops/tidying/`:
1. `integration_test.md`
2. `reference_check.md`
3. `stub_scan.md`
4. `dedup.md`
5. `cleanup.md`

## Template Structure

Each template follows this format:

```markdown
---
---

# <Name> Hygiene Check

## Scope
What to check

## Commands
Scripts/tools to run

## Output Schema
{
  "findings": [...],
  "auto_fixable": [...],
  "requires_review": [...]
}

## Auto-Fix Rules
When can this Drop auto-fix vs escalate
```

## Templates

### 1. integration_test.md
- Runs INTEGRATION_TESTS.json if exists
- Checks cross-component behavior
- **auto_fix: false** (report only)

### 2. reference_check.md  
- Find broken imports (pylint, tsc)
- Find dead file references
- Find invalid paths in configs
- **auto_fix: true** for unused imports

### 3. stub_scan.md
- Find TODO/FIXME comments
- Find `NotImplementedError`
- Find placeholder strings
- Find empty function bodies
- **auto_fix: false** (escalate - indicates incomplete work)

### 4. dedup.md
- Find duplicate code (>10 lines)
- Find redundant files
- **auto_fix: partial** (suggest refactoring)

### 5. cleanup.md
- Find console.log/print debug statements
- Find commented-out code (>3 lines)
- Find unused variables
- **auto_fix: true** for safe removals

## Success Criteria

- [ ] All 5 templates created
- [ ] Each has clear scope, commands, output schema
- [ ] Auto-fix rules are explicit
- [ ] Templates are valid markdown with frontmatter

## Notes for Orchestrator

- D4.2 (Tidying Orchestration) uses these templates
- Templates define WHAT to check; D4.2 defines HOW to orchestrate
