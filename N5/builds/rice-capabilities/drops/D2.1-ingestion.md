# Drop Brief: Ingestion — Handler Framework & Channels

**Mission:** Build the Ingestion capability — the mailroom that receives and classifies inbound content from all channels (email, voice, webhook, Zo2Zo), assigns UUID/timestamp/classification, and routes to the appropriate employee.

## Context

Wave 2 of rice-capabilities. Ingestion depends on Security (D1.1) for the inbound gate and audit logging, and Memory (D1.2) for conversation logging. Every inbound item flows: receive → security gate → classify → route → log.

Ingestion is the "mailroom" — it doesn't handle messages itself, it receives, stamps, classifies, and routes them to the right employee. The routing resolver that maps channel+intent to employee name will be built in rice-integration — for now, ingestion prepares the data and returns a routing recommendation.

This is Layer 1 — the classifier uses keyword matching, not LLM. Channel handlers are skeletons with the framework in place; Layer 2 adds real integrations (VAPI, Gmail tags, etc.).

## Scenarios

S1: Base handler processes inbound item end-to-end

  When: `email_handler.receive({"subject": "[JD] Senior Engineer", "from": "recruiter@example.com", "body": "Please review this candidate"})` is called
  Then: Returns an InboundItem dict with keys: id (UUID), timestamp, channel ("email"), classification (dict with type/urgency/topic), routing_recommendation (str), raw_content, security_result

S2: Classifier assigns type and urgency

  When: `classify.classify_content("URGENT: Server is down, need help immediately")` is called
  Then: Returns `{"type": "request", "urgency": "high", "topics": ["server", "help"]}`
  Verify: `python3 -c "from Zoffice.capabilities.ingestion.classify import classify_content; r = classify_content('URGENT: Server is down, need help immediately'); assert r['urgency'] in ('high', 'critical'); print('PASS')"`

S3: Email handler extracts tags from subject
  Given: An email with subject "[KNOWLEDGE] API Documentation Update"
  When: `email_handler.receive({"subject": "[KNOWLEDGE] API Documentation Update", "from": "team@example.com", "body": "Updated docs"})` is called
  Then: The routing_recommendation is based on the [KNOWLEDGE] tag pattern in config

## Requirements

### base.py (Base Handler)
- Abstract class `InboundHandler` with methods: `receive(raw_input) -> InboundItem`, `classify(item)`, `route(item)`, `log(item)`
- `InboundItem` is a dict with keys: id (uuid4), timestamp (ISO), channel (str), raw_content, classification, routing_recommendation, security_result, metadata
- Base `receive()` flow: generate UUID → security gate → classify → route → audit log → return
- All handlers extend this and override channel-specific parsing

### email_handler.py
- Extends InboundHandler, channel="email"
- Parses subject for tags: `[TAG]` pattern → extracted as list
- Tag-based routing: reads ingestion/config.yaml for tag→employee mappings
- Falls back to channel default from routing.yaml if no tag match

### voice_handler.py
- Extends InboundHandler, channel="voice"
- Skeleton: accepts dict with caller_phone, caller_name (optional), transcript (optional)
- Always routes to channel default (receptionist) in Layer 1
- Designed to be extended by VAPI webhook integration in Layer 2

### webhook_handler.py
- Extends InboundHandler, channel="webhook"
- Accepts dict with source (str), payload (dict), signature (optional)
- Source-based routing: reads config for source→employee mappings

### zo2zo_handler.py
- Extends InboundHandler, channel="zo2zo"
- Accepts dict with source_instance, api_key (optional), content, metadata
- Trust check: reads Zoffice/config/security.yaml trusted_instances list
- Trusted → normal routing. Untrusted → flags item, still processes

### classify.py
- `classify_content(text: str) -> dict` with keys: type, urgency, topics
- Types: inquiry, request, complaint, information, escalation
- Urgency: low, normal, high, critical
- Keyword-based for v1 (not LLM). Patterns in config.yaml.

### config.yaml
- `channels`: per-channel settings (enabled, default_employee)
- `tag_routing`: tag→employee mappings for email
- `source_routing`: source→employee mappings for webhooks
- `classification`: keyword patterns for type and urgency detection

## Anti-Patterns (Avoid)

- Don't use LLM for classification in v1 — keyword matching only
- Don't implement actual Gmail/VAPI integration — these are skeletons
- Don't modify routing.yaml — just read from it
- Don't duplicate security gate logic — import and call inbound_gate.validate()
- Don't skip the security gate for any channel, including zo2zo from trusted sources
