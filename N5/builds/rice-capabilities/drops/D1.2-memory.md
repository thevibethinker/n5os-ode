# Drop Brief: Memory — DB Helpers & Managers

**Mission:** Build the Memory capability — the office's persistent storage layer providing CRUD operations for contacts, conversations, and decisions via helper functions over the shared DuckDB database.

**Output:**
- `Zoffice/capabilities/memory/config.yaml`
- `Zoffice/capabilities/memory/db_helpers.py`
- `Zoffice/capabilities/memory/contact_manager.py`
- `Zoffice/capabilities/memory/conversation_logger.py`
- `Zoffice/capabilities/memory/decision_queue.py`
- `Zoffice/capabilities/memory/README.md` (update existing)

---

## Context

This is Wave 1 of the rice-capabilities build. Memory is foundational because all other capabilities store and retrieve data through it — no raw SQL outside this module.

The database `Zoffice/data/office.db` already exists with 5 tables: audit, contacts, decisions, conversations, evaluations. The schemas are already created by rice-core. This Drop wraps them with clean Python helper functions.

Memory is the "filing cabinet and Rolodex" — it provides structured access to office data. Employees never touch the database directly; they call Memory's helper functions.

**Important:** The audit table is managed by the Security capability (D1.1), not Memory. Memory manages: contacts, conversations, decisions, and evaluations. You should call the audit writer from Security for logging your own operations, but since D1.1 runs in parallel, handle the case where the audit writer may not exist yet — log a warning and continue.

## Scenarios

S1: Create and retrieve a contact
  Given: An empty contacts table in office.db
  When: `contact_manager.upsert_contact(name="Jane Doe", email="jane@example.com", phone="+15551234567")` is called, then `contact_manager.find_contact(email="jane@example.com")` is called
  Then: The second call returns a dict with name="Jane Doe", email="jane@example.com", phone="+15551234567"
  Verify: `python3 -c "from Zoffice.capabilities.memory.contact_manager import upsert_contact, find_contact; upsert_contact(name='Jane Doe', email='jane@example.com', phone='+15551234567'); r = find_contact(email='jane@example.com'); assert r['name'] == 'Jane Doe'; print('PASS')"`

S2: Decision queue lifecycle (create → resolve)
  Given: An empty decisions table
  When: `decision_queue.create_decision(summary="Should we send the report?", origin_employee="chief-of-staff")` returns an ID, then `decision_queue.resolve_decision(id, resolution="Yes, send it", resolved_by="human")` is called
  Then: `decision_queue.get_pending_decisions()` returns an empty list; the resolved decision has status="resolved"
  Verify: `python3 -c "from Zoffice.capabilities.memory.decision_queue import create_decision, resolve_decision, get_pending_decisions; did = create_decision(summary='Test decision', origin_employee='cos'); resolve_decision(did, resolution='Approved', resolved_by='human'); assert len(get_pending_decisions()) == 0; print('PASS')"`

S3: Conversation logging
  Given: An empty conversations table
  When: `conversation_logger.log_conversation(channel="email", employee="receptionist", counterparty_id="c-123", summary="Greeting and routing")` is called
  Then: `conversation_logger.get_conversations(employee="receptionist")` returns a list containing the logged conversation
  Verify: `python3 -c "from Zoffice.capabilities.memory.conversation_logger import log_conversation, get_conversations; log_conversation(channel='email', employee='receptionist', counterparty_id='c-123', summary='Test'); r = get_conversations(employee='receptionist'); assert len(r) >= 1; print('PASS')"`

S4: Contact upsert is idempotent
  Given: A contact with email="jane@example.com" already exists
  When: `contact_manager.upsert_contact(name="Jane Doe-Smith", email="jane@example.com")` is called
  Then: The contact's name is updated to "Jane Doe-Smith" and no duplicate is created
  Verify: `python3 -c "import duckdb; conn = duckdb.connect('Zoffice/data/office.db'); count = conn.execute(\"SELECT count(*) FROM contacts WHERE email='jane@example.com'\").fetchone()[0]; assert count == 1; print('PASS')"`

S5: DB connection handles missing database gracefully
  Given: No database file at a specified path
  When: `db_helpers.get_db(db_path="/tmp/nonexistent.db")` is called
  Then: A new database is created (DuckDB creates files on connect) — no crash
  Verify: `python3 -c "from Zoffice.capabilities.memory.db_helpers import get_db; conn = get_db(db_path='/tmp/test_memory.db'); assert conn is not None; print('PASS')"`

## Requirements

### db_helpers.py
- `get_db(db_path=None) -> duckdb.DuckDBPyConnection` — singleton connection, default path from config.yaml
- Connection lifecycle management (thread-safe if possible, at minimum safe for sequential use)
- Generic `execute_query(query, params=None)` for internal use by other memory modules

### contact_manager.py
- `upsert_contact(name, email=None, phone=None, organization=None, relationship=None, tags=None, profile=None) -> str` — returns contact ID
- `find_contact(email=None, phone=None, name=None) -> dict | None` — lookup by any field
- `get_contact(id) -> dict | None` — direct lookup
- `update_contact(id, **fields) -> bool` — partial update
- `list_contacts(relationship=None, tag=None, limit=50) -> list[dict]`
- Upsert logic: if email OR phone matches existing contact, update. Otherwise create new.

### conversation_logger.py
- `log_conversation(channel, employee, counterparty_id=None, summary=None, duration_seconds=None, satisfaction=None, metadata=None) -> str` — returns conversation ID
- `get_conversations(counterparty_id=None, employee=None, channel=None, since=None, limit=50) -> list[dict]`
- `end_conversation(id, summary=None, satisfaction=None) -> bool` — updates ended_at and optional fields

### decision_queue.py
- `create_decision(summary, origin_employee=None, full_context=None, options=None, recommendation=None) -> str` — returns decision ID
- `get_pending_decisions(employee=None) -> list[dict]` — all with status="pending"
- `resolve_decision(id, resolution, resolved_by) -> bool` — sets status="resolved", resolved_at, resolution, resolved_by
- `expire_decisions(older_than_days=30) -> int` — sets status="expired" for old pending decisions, returns count

### config.yaml
- `db_path`: path to office.db (default: `Zoffice/data/office.db`)
- `retention`: settings for data retention
- `defaults`: default values for optional fields

## Anti-Patterns (Avoid)

- Don't expose raw SQL to callers — all SQL stays inside this module
- Don't create or modify database tables — they already exist
- Don't import from Security or other capabilities — if you need audit logging, make it optional (try/except import)
- Don't return DuckDB internal types — always convert to plain dicts/lists
- Don't use connection pooling — single DuckDB connection is sufficient for single-instance
