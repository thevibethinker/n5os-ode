# Drop 4.1: Localization Framework

## Objective
Design the protocol for adapting archetype patterns to individual client contexts. This is a RESEARCH drop—zoputer needs to think through the philosophy and create guidelines.

## Research Questions

1. **Decision Authority**: When should zoputer...
   - Auto-adapt a pattern to client context?
   - Ask V for guidance?
   - Ask the client for clarification?

2. **Context Categories**: What dimensions need localization?
   - Industry/sector differences
   - Team size (solo vs. startup vs. enterprise)
   - Technical sophistication
   - Use case priorities
   - Cultural/communication preferences

3. **Adaptation Patterns**: What transforms are safe?
   - Changing folder names to match client's mental model
   - Adding/removing steps based on team size
   - Swapping tools (e.g., Notion vs. Airtable)
   - Adjusting automation frequency

4. **Preservation Boundaries**: What must stay consistent?
   - N5 naming conventions
   - Audit logging requirements
   - Security gate rules
   - Version control practices

```markdown
## Questionnaire

Initial client intake questions to determine localization needs:
1. Team size?
2. Primary industry?
3. Technical comfort (1-5)?
4. Current tools (list)?
5. Biggest friction point?
6. Time availability for setup?

## Escalation Triggers

Auto-escalate to V when:
- Request involves Personal/ or Zo/ equivalent
- Client asks for API keys or secrets
- Pattern would violate security gate rules
- Unclear how to adapt (confidence < 0.7)
```
