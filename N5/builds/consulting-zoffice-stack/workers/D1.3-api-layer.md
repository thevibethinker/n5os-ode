# Drop 1.3: API Communication Layer

## Objective
Build the direct API communication channel between va.zo.computer and zoputer.zo.computer using /zo/ask. This is a trusted internal channel—no security overhead, just reliable transport with full audit logging.

## Requirements

### API Key Setup (Manual - V must do this)
1. On va.zo.computer: [Settings > Developers] → Create key `ZO_ARCHETYPE_SYNC`
2. On zoputer.zo.computer: [Settings > Developers] → Create key `ZO_CONSULTING_API`
3. Store both keys securely (environment variables only)

### va-side Client (`N5/scripts/zoputer_client.py`)
CLI interface:
```bash
python3 N5/scripts/zoputer_client.py export \
  --skill skills/content-classifier \
  --version 1.2.3 \
  --notes "Added tier detection"

python3 N5/scripts/zoputer_client.py ping

python3 N5/scripts/zoputer_client.py status
```

Features:
- Bundle skill directory into tarball
- Generate manifest.json with metadata
- POST to zoputer via /zo/ask
- Retry logic (3 attempts, exponential backoff)
- Timeout: 30 seconds
- Log to audit system (Drop 1.2)

### zoputer-side Endpoint
A route on zoputer.zo.computer that accepts:
```json
{
  "action": "ingest_skill",
  "skill_name": "content-classifier",
  "version": "1.2.3",
  "bundle_url": "temporary_signed_url",
  "manifest": {...},
  "source": "va.zo.computer",
  "timestamp": "2026-02-06T..."
}
```

Response:
```json
{
  "status": "success|error",
  "ingested_files": ["..."],
  "errors": [],
  "zoputer_timestamp": "..."
}
```

### Error Handling
- Network timeout: Retry 3x, then queue for next attempt
- Authentication failure: Alert V immediately
- Ingestion failure: Log details, notify V
- Bundle corruption: Request re-send

### Health Check
`zoputer_client.py ping` returns:
- Response time
- zoputer version/status
- Last successful export timestamp
