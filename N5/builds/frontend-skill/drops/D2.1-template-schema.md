# D2.1: Template Schema Design

## Context

The template system lets us capture "design DNA" from successful pages and reliably reproduce that aesthetic with different content. This is the schema that defines what a template contains.

**Key design tension:** Templates need to be specific enough to produce consistent results, but flexible enough to work with different content.

## MUST DO

1. **Create `Skills/frontend/templates/_TEMPLATE.yaml`** — The format specification

2. **Schema must include:**

   ```yaml
   # Required metadata
   name: template-slug  # lowercase, hyphens only
   description: |
     What this template is good for. When to use it.
   source: |
     URL of the original page, or description of where this came from.
   created: YYYY-MM-DD
   
   # Color palette (all required)
   palette:
     primary: "#hex"      # Main brand color
     secondary: "#hex"    # Supporting color
     accent: "#hex"       # Highlights, CTAs
     background: "#hex"   # Page background
     surface: "#hex"      # Card/section backgrounds
     text: "#hex"         # Primary text
     text_muted: "#hex"   # Secondary text
   
   # Typography
   typography:
     heading_font: "Font Name"  # Google Fonts or system font
     body_font: "Font Name"
     mono_font: "Font Name"     # For code, optional
     scale: tight | normal | loose  # Overall spacing
     heading_weight: 400-900    # Font weight for headings
   
   # Layout
   layout:
     max_width: narrow | standard | wide | full
     # narrow: 768px, standard: 1024px, wide: 1280px, full: 100%
     spacing: compact | balanced | airy
     # compact: 1-2rem sections, balanced: 3-4rem, airy: 5-6rem
     border_radius: none | subtle | rounded | pill
     # none: 0, subtle: 4px, rounded: 8-12px, pill: 9999px
   
   # Sections (ordered list of what the page contains)
   sections:
     - type: hero | features | testimonials | pricing | cta | faq | stats | team | logos
       variant: string  # Optional variant name for the section type
   
   # Style notes (freeform, critical for capturing nuance)
   style_notes: |
     Freeform description of:
     - The overall aesthetic feel
     - Specific techniques used (shadows, borders, gradients)
     - What makes this template distinctive
     - Things that are hard to capture in structured fields
   
   # Optional: Reference images
   reference_images:
     - path: /path/to/image.png
       description: What this shows
   ```

3. **Add documentation within the file** explaining:
   - Each field's purpose
   - Valid values for enum fields
   - How the template will be applied during generation

4. **Design for extensibility** — Include a `custom` field for template-specific values:
   ```yaml
   custom:
     key: value  # Template-specific overrides
   ```

2. Has clear comments explaining each field

5. Balances structure with flexibility (style_notes for nuance)
