# Drop Brief: Forward Broadcast

**Mission:** Add a `broadcast` field to deposits that Pulse collects and injects into subsequent Drop briefs, enabling forward-only inter-Drop communication.

**Output:** Modified `pulse.py` with broadcast collection and injection logic.

---

## Context

This is part of a Pulse self-upgrade build. We're adding features inspired by Claude Agent Teams, where teammates can share findings with each other.

**Key insight:** We want forward-only communication (Drop A tells Drop B something), NOT peer-to-peer messaging. This keeps the isolation benefits while adding information propagation.

**How it works:**
1. Drop completes and includes `broadcast` field in deposit JSON
2. Pulse collects all broadcasts from completed Drops
3. When spawning a new Drop, Pulse injects a "## Broadcasts from Prior Drops" section into the brief

## Requirements

### 1. Deposit Schema Extension

Deposits MAY include an optional `broadcast` field:

```json
{
  "drop_id": "D1.1",
  "status": "complete",
  "broadcast": "Auth tokens expire after 30 min—all Drops should use short-lived tokens",
  "summary": "...",
  "artifacts": []
}
```

- `broadcast` is optional (backward compatible)
- Single string, not array (keep it simple)
- Max ~500 chars recommended (injected into briefs, don't bloat)

### 2. Broadcast Collection

In `tick()` or wherever deposits are processed:

```python
def collect_broadcasts(slug: str) -> list[dict]:
    """Collect all broadcasts from completed deposits."""
    broadcasts = []
    deposits_dir = BUILDS_DIR / slug / "deposits"
    for deposit_path in deposits_dir.glob("*.json"):
        if "_filter" in deposit_path.name:
            continue
        with open(deposit_path) as f:
            deposit = json.load(f)
        if deposit.get("broadcast"):
            broadcasts.append({
                "drop_id": deposit.get("drop_id"),
                "broadcast": deposit["broadcast"],
                "timestamp": deposit.get("timestamp")
            })
    return broadcasts
```

### 3. Broadcast Injection

When spawning a Drop (in `spawn_drop()` or equivalent):

```python
def inject_broadcasts(brief_content: str, broadcasts: list[dict]) -> str:
    """Inject broadcasts section into brief before spawning."""
    if not broadcasts:
        return brief_content
    
    section = "\n\n## Broadcasts from Prior Drops\n\n"
    section += "These findings were shared by earlier Drops in this build:\n\n"
    for b in broadcasts:
        section += f"- **{b['drop_id']}:** {b['broadcast']}\n"
    
    # Insert before "## Requirements" or "## Context" or at end
    for marker in ["## Requirements", "## Context", "## Files to Read"]:
        if marker in brief_content:
            return brief_content.replace(marker, section + marker)
    
    return brief_content + section
```

### 4. Persist Broadcasts in meta.json

Store collected broadcasts in `meta.json` under `broadcasts` key:

```json
{
  "broadcasts": [
    {"drop_id": "D1.1", "broadcast": "...", "timestamp": "..."}
  ]
}
```

This allows inspection and survives orchestrator restarts.

### 5. Integration Points

Modify these functions in `pulse.py`:
- `tick()` — After processing a deposit, check for broadcast and add to meta
- `spawn_drop()` or equivalent — Before spawning, inject broadcasts into brief

## Anti-Patterns (Avoid)

- Don't add bidirectional messaging — this is forward-only by design
- Don't make `broadcast` required — it's optional
- Don't inject broadcasts into the DROP that created them (only subsequent Drops)
- Don't use learnings system — broadcasts are build-scoped, learnings are system-scoped
