# Drop Brief: Dynamic Task Claiming

**Mission:** Add a task pool system where Drops can self-assign work from a shared pool, enabling more efficient parallel execution.

**Output:** Modified `pulse.py` with task pool logic and atomic claiming.

---

## Context

This is part of a Pulse self-upgrade build. Currently, all Drops are assigned upfront. This is inefficient when:
- Some Drops finish faster than others
- The pool has many small, uniform tasks
- You want to maximize parallelism without pre-planning every assignment

**Example use case:** Processing 20 similar items (enrich profiles, validate records, etc.). Instead of creating D1.1-D1.20, create a pool and let 4 workers claim from it dynamically.

## Requirements

### 1. Meta.json Schema Extension

Add optional `task_pool` section:

```json
{
  "slug": "enrich-profiles",
  "task_pool": {
    "enabled": true,
    "tasks": [
      {"id": "T001", "type": "enrich", "target": "profile_1.json", "status": "pending"},
      {"id": "T002", "type": "enrich", "target": "profile_2.json", "status": "pending"},
      {"id": "T003", "type": "enrich", "target": "profile_3.json", "status": "pending"}
    ],
    "max_concurrent_claims": 4,
    "worker_drops": ["D1.1", "D1.2", "D1.3", "D1.4"]
  }
}
```

- `enabled: true` — Activate pool mode
- `tasks` — Array of pool tasks with IDs and status
- `max_concurrent_claims` — How many tasks can be claimed at once (defaults to len(worker_drops))
- `worker_drops` — Which Drops can claim from the pool

### 2. Task States

Pool tasks have these statuses:
- `pending` — Available for claiming
- `claimed` — Assigned to a Drop
- `complete` — Finished successfully
- `failed` — Failed, may be re-claimable

### 3. Atomic Claiming

**Critical:** Multiple Drops running in parallel could race to claim the same task. Use file locking:

```python
import fcntl

def claim_task(slug: str, drop_id: str) -> dict | None:
    """Atomically claim a task from the pool."""
    meta_path = BUILDS_DIR / slug / "meta.json"
    
    with open(meta_path, "r+") as f:
        # Acquire exclusive lock
        fcntl.flock(f.fileno(), fcntl.LOCK_EX)
        try:
            meta = json.load(f)
            pool = meta.get("task_pool", {})
            
            if not pool.get("enabled"):
                return None
            
            # Find first pending task
            for task in pool.get("tasks", []):
                if task["status"] == "pending":
                    task["status"] = "claimed"
                    task["claimed_by"] = drop_id
                    task["claimed_at"] = datetime.now(timezone.utc).isoformat()
                    
                    # Write back
                    f.seek(0)
                    f.truncate()
                    json.dump(meta, f, indent=2)
                    
                    return task
            
            return None  # Pool exhausted
        finally:
            fcntl.flock(f.fileno(), fcntl.LOCK_UN)
```

### 4. Pool Worker Brief Template

Workers claiming from a pool need a different brief structure:

```markdown
---
drop_id: D1.1
pool_worker: true
---

# Pool Worker: Enrich Profiles

You are a pool worker. Your job:
1. Claim a task: Run claim logic (injected by Pulse)
2. Execute the task
3. Mark complete and claim another
4. Repeat until pool is exhausted
5. Deposit summary of all completed tasks

## Task Execution

For each claimed task:
- Read target file
- Perform enrichment
- Write result
- Mark task complete

## On Pool Exhaustion

When no tasks remain, deposit with summary of all completed work.
```

### 6. Integration with Spawn

Modify spawn logic:
- If Drop has `pool_worker: true` in brief, inject claiming instructions
- Don't spawn pool workers until pool is non-empty
- Pool workers can be spawned in parallel up to `max_concurrent_claims`

### 7. Completion Handling

When a pool worker deposits:
- Update the claimed task's status to `complete` (or `failed`)
- If pool still has pending tasks, worker can re-claim (in same session, or re-spawn)
- If pool exhausted, mark worker Drop as complete

### 8. Edge Cases

- **Pool exhausted mid-run:** Worker exits gracefully
- **Worker fails without depositing:** Task stays `claimed`; add `claim_timeout` to auto-release (default 30 min)
- **All workers done but tasks remain:** Surface in STATUS.md as warning

## Anti-Patterns (Avoid)

- Don't use database for locking — file lock is sufficient and simpler
- Don't require pool for all builds — it's optional
- Don't auto-retry failed tasks — that's a policy decision for the build author
- Don't spawn unlimited workers — respect `max_concurrent_claims`
