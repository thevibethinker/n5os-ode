# Drop Brief: Foundation — Config, Schema, Migration

**Mission:** Create the understanding_bank.json knowledge tracker, migrate existing glossary data into it, and update the Pulse config to reference the new learning_mode system.

**Output:**
- `N5/config/understanding_bank.json` (CREATE)
- `N5/scripts/migrate_glossary.py` (CREATE)
- `Skills/pulse/config/pulse_v2_config.json` (UPDATE)

---

## Context

We're replacing the old regex-based teaching system with an LLM-native learning mode. The first step is creating the data foundation: a single `understanding_bank.json` that consolidates V's technical knowledge tracking (previously split across a glossary file and inline persona text). This file becomes the single source of truth for V's technical understanding level.

The existing glossary at `N5/pulse/teaching/glossary.json` has 4 terms tracked. These need to be migrated. The Teacher persona's inline Technical Calibration (domain-level assessment) also needs to be captured here.

## Requirements

### understanding_bank.json Schema

```json
{
  "version": "1.0",
  "created": "2026-02-17",
  "last_updated": "2026-02-17",
  "meta": {
    "description": "V's technical understanding tracker. Single source of truth for concept mastery.",
    "migrated_from": "N5/pulse/teaching/glossary.json"
  },
  "domains": {
    "system_architecture": { "level": "solid", "notes": "workflows, SSOT, modular design" },
    "data_structures": { "level": "solid", "notes": "JSONL, schemas, file organization" },
    "abstractions": { "level": "solid", "notes": "APIs as contracts, state management" },
    "implementation": { "level": "learning", "notes": "async/await, error handling, HTTP mechanics" },
    "developer_tooling": { "level": "learning", "notes": "git workflows, debugging, testing" },
    "low_level": { "level": "new", "notes": "memory, concurrency, networking internals" }
  },
  "concepts": []
}
```

Concept entry schema:
```json
{
  "term": "string",
  "precise_name": "string",
  "level": "new|encountered|learning|familiar|solid|deep",
  "v_description": "string (V's own words)",
  "first_encountered": "ISO date",
  "last_engaged": "ISO date",
  "evidence": ["array of build/conversation references"],
  "domain": "string matching domains key"
}
```

### Migration Script

- Read glossary.json
- For each term: map `absorbed: true` → `level: "solid"`, `absorbed: false` → `level: "encountered"`
- Preserve: `v_description`, `first_encountered`, `usage_count` (into evidence notes)
- Map terms to domains (MECE → system_architecture, idempotent → implementation, etc.)
- Write to `N5/config/understanding_bank.json`
- Print summary of migrated terms

### Config Update

Replace the `vibe_teacher` key in pulse_v2_config.json with:
```json
"learning_mode": {
  "enabled": true,
  "default_build_mode": "learning",
  "understanding_bank_path": "N5/config/understanding_bank.json",
  "friction_levels": {
    "minimal": "manual spawn, summary only",
    "standard": "manual spawn, decision points, wave reviews",
    "full": "manual spawn, decision points, wave reviews, learning drops, pedagogical AAR"
  },
  "default_friction": "standard"
}
```

## Success Criteria

- [ ] `N5/config/understanding_bank.json` exists with valid JSON

- [ ] Each concept has all required fields (term, precise_name, level, domain, first_encountered)

## Anti-Patterns (Avoid)

- Don't modify glossary.json — it's being migrated FROM, not TO
- Don't delete the teaching/ directory — that's D1.2's job
- Don't overcomplicate the understanding_bank schema — start simple, version field enables future evolution
