# Drop Brief: AuthKit integration (Hono/Bun)

**Mission:** Integrate WorkOS AuthKit into Where’s V so the entire site is gated behind login.

**Primary codebase:** `Sites/wheres-v-staging`

**Output (artifacts):**
- UPDATE `Sites/wheres-v-staging/server.ts` — add WorkOS AuthKit routes + session + auth middleware
- (Optional) UPDATE `Sites/wheres-v-staging/README.md` — add env var requirements section (no secrets)

---

## Context

V wants Where’s V to be private:
- Entire site requires login
- Invite-only access: users cannot freely self-sign-up
- Enforcement should be in two layers:
  1) WorkOS AuthKit: disable sign-up / use invitations
  2) App-level allowlist: only pre-authorized emails may access even if they manage to authenticate

This site runs **Bun + Hono** (`server.ts`) and serves a SPA from `./dist`.

WorkOS docs for sessions describe:
- `authenticateWithCode({ session: { sealSession: true, cookiePassword }})`
- store `sealedSession` in a cookie (httpOnly, secure, sameSite)
- `loadSealedSession(...).authenticate()` and `refresh()` for middleware protection

---

## Requirements

### Environment variables (read from `process.env`)
- `WORKOS_API_KEY`
- `WORKOS_CLIENT_ID`
- `WORKOS_COOKIE_PASSWORD` (must be 32 chars)
- `WORKOS_REDIRECT_URI` (e.g., `https://<PUBLIC_DOMAIN>/callback`)

### Routes to implement
- `GET /login`
  - Redirect to WorkOS AuthKit authorization URL (provider: `authkit`)
  - Use `WORKOS_REDIRECT_URI`
- `GET /callback`
  - Read `code` query param
  - Call `workos.userManagement.authenticateWithCode({ clientId, code, session: { sealSession: true, cookiePassword } })`
  - Set cookie `wos-session` to `sealedSession` with: `path=/`, `httpOnly=true`, `secure=true`, `sameSite=lax`
  - Redirect to `/`
- `GET /logout`
  - Load sealed session; get logout URL; clear cookie; redirect
  - If no session, just clear cookie and redirect to `/login`

### Auth middleware (global gate)
- Must gate **everything**, including the SPA HTML, except a small allowlist of routes/assets.
- Permit unauthenticated access to:
  - `/login`, `/callback`, `/logout`, `/health`
  - static asset requests (e.g. `/assets/*` and typical extensions like `.js`, `.css`, `.svg`, `.png`, `.ico`, `.map`)
- For all other requests:
  - Attempt `loadSealedSession` and `session.authenticate()`
  - If `authenticated`: continue
  - If missing cookie or invalid session: redirect to `/login`
  - If session not authenticated but refreshable: `session.refresh()` then update cookie and redirect back to same URL

### Invite-only allowlist enforcement
- After `session.authenticate()` yields `user`, check `user.email` is in `Sites/wheres-v-staging/config/allowed_emails.json`.
- If not allowed:
  - Clear cookie
  - Return `403` with a short message (or redirect to `/login?error=not_authorized`)

### Bun/Hono specifics
- Use Hono cookie helpers (e.g., `getCookie`, `setCookie`, `deleteCookie` from `hono/cookie`) instead of Express cookie-parser.
- Keep existing `/api/*` endpoints working but gated.

## Success Criteria

- [ ] Unauthenticated request to `/` redirects to `/login`
- [ ] Unauthenticated request to `/api/current-state` redirects to `/login`
- [ ] After completing AuthKit flow, session cookie is set and `/` loads
- [ ] If authenticated user email is not in allowlist, access is denied and cookie is cleared
- [ ] Code compiles (TypeScript) and server starts

## Anti-Patterns (Avoid)
- Don’t add secrets to the repo.
- Don’t introduce an additional auth library (keep it WorkOS-only).
- Don’t make the SPA depend on localStorage tokens; use httpOnly cookies.
