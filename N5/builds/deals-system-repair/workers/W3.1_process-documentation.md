# Worker Brief: Process Documentation

**Pre-decided thread title:** `[deals-system-repair] W3.1: Process Documentation`

## Purpose
Document the classification rules and broker linkage protocols that were implicit but caused data issues. This ensures future data entry and sync processes follow consistent rules.

## Context
Wave 1-2 fixed scripts and synced data. V's review session revealed **process gaps**:
1. Calendly was classified as "leadership" when it should be "acquirer" (company that might buy Careerspan)
2. Broker→acquirer linkages (Ivor→Handshake, Vir→Handshake) weren't captured from meetings

These happened because the rules weren't documented.

## Deliverables

### 1. Create `N5/docs/deal-classification-rules.md`

Content should include:

**Deal Types:**
- `zo_partnership` — Company/person for Zo partnerships
- `careerspan_acquirer` — Company that might acquire Careerspan (the TARGET)
- `leadership` — Person relationship for Careerspan deals where no company context exists yet

**Classification Rules:**
1. If a company could acquire Careerspan → `careerspan_acquirer`, even if entry point is a person
2. Person at acquirer company → link to acquirer deal via notes/leader field, NOT separate leadership deal
3. `leadership` is only for people without clear company affiliation OR people being cultivated for future unknown opportunities
4. One deal per company, multiple contacts can be linked

**Examples:**
- Calendly (company) + Tope (founder) → ONE acquirer deal `cs-acq-calendly` with Tope as primary contact
- Random exec met at conference, no company target yet → `leadership` deal
- Korn Ferry contact for intros → `careerspan_acquirer` for Korn Ferry with contact as liaison

### 2. Document Broker Linkage Protocol

Add section to classification rules OR create `N5/docs/broker-linkage-protocol.md`:

**When meetings mention intro offers:**
1. Identify pattern: "[Person] can/will introduce to [Company]"
2. Person is a broker → Update `deal_contacts.angle_strategy` with target company name(s)
3. Person is a lead → Update their notes field with intro potential
4. Create/verify acquirer deal exists for target company

**Source of truth:** Meetings reveal intros → Update contact records → Reflect in deal tables

### 3. Add Comments to Config (optional)

If helpful, add clarifying comments to `N5/config/notion_field_mapping.json` header explaining which Notion database maps to which deal type.
