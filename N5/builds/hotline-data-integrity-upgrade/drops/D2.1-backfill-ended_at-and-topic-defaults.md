# D2.1: Backfill ended_at and topic defaults

## Mission

Run a safe backfill after forward ingestion is fixed:
- update historical rows where `ended_at = started_at` and `duration_seconds > 0`,
- set `topics_discussed` fallback for null/empty rows,
- produce per-dataset before/after counts for auditability.

## Context

Backfill scope:
- `Datasets/zo-hotline-calls/data.duckdb`
- `Datasets/career-hotline-calls/data.duckdb`
- `Datasets/vibe-pill-calls/data.duckdb`

Do not backfill `Datasets/vapi-calls/data.duckdb` unless explicitly requested.
Backfill must be idempotent and predicate-constrained.

## Scenarios

S1: Targeted timestamp backfill
Given an existing row with `ended_at = started_at` and `duration_seconds=180`
When backfill runs
Then row `ended_at` becomes `started_at + 180s`
Verify row no longer appears in anomaly query.

S2: Do-not-touch valid rows
Given a row where `ended_at > started_at` and duration already aligned
When backfill runs
Then row remains unchanged
Verify updated row count excludes valid rows.

S3: Topic fallback backfill
Given rows with null/blank `topics_discussed`
When backfill runs
Then value is set to service fallback (`general`/`general_advisory`)
Verify no null/blank topics remain in targeted datasets.

## Affected Files

- `N5/builds/hotline-data-integrity-upgrade/artifacts/hotline_backfill.sql` or script equivalent
- `N5/builds/hotline-data-integrity-upgrade/artifacts/hotline_backfill_report.json`
- `Datasets/zo-hotline-calls/data.duckdb`
- `Datasets/career-hotline-calls/data.duckdb`
- `Datasets/vibe-pill-calls/data.duckdb`

## Success Criteria

- [ ] Backfill updates only rows matching explicit anomaly predicates.
- [ ] Before/after audit counts are captured per dataset.
- [ ] Re-running backfill yields zero or minimal net changes (idempotent behavior).
