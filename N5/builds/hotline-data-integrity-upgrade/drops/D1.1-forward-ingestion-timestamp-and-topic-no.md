# D1.1: Forward ingestion timestamp and topic normalization

## Mission

Patch forward ingestion logic so new `calls` rows have consistent timestamps and topic defaults:
- compute `ended_at` from `started_at + duration_seconds` when webhook payload omits/duplicates end time,
- preserve explicit `ended_at` when valid,
- normalize empty topic values to canonical fallback label.

## Context

Current bug pattern: many rows have `ended_at = started_at` while `duration_seconds > 0`.
This appears in:
- `Skills/zoren-hotline/scripts/hotline-webhook.ts`
- `Skills/zo-hotline/scripts/hotline-webhook.ts`
- `Skills/career-coaching-hotline/scripts/hotline-webhook.ts`
- logger helpers where call insert payload defaults `ended_at` to `now`.
Do not modify unrelated behavior (billing, escalation, email follow-up).

## Scenarios

S1: Missing ended_at with valid duration
Given a call payload has `startedAt` and `durationSeconds=120` but no `endedAt`
When the webhook logs the call
Then stored `ended_at` equals `started_at + 120s`
Verify DB query shows `DATEDIFF('second', started_at, ended_at)` within +/-2 seconds of `duration_seconds`.

S2: Ended_at equals started_at from upstream
Given payload has identical `startedAt` and `endedAt` with `durationSeconds=75`
When logging occurs
Then code overrides to computed end timestamp
Verify persisted row no longer has zero-span interval.

S3: Topic classifier yields empty output
Given async topic classification returns empty/invalid topic list
When call row is inserted/updated
Then `topics_discussed` is set to normalized fallback (`general_advisory`/`general`)
Verify no newly inserted rows have null/empty `topics_discussed`.
