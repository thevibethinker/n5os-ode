# D1.2: Purge Stale Resources & Orphaned Data

## Objective

Clean both `vectors_v2.db` and `brain.db` of stale resources (pointing to non-existent files), orphaned resources (no blocks), and orphaned blocks (no vectors). This reduces index size, improves search quality (no dead results), and reduces brute-force scan time until ANN index is active.

## Context

### Current State (from audit)
**vectors_v2.db:**
- Total resources: 9,683
- Stale (file missing): 2,504 (25.9%)
- Resources without blocks: 574

**brain.db:**
- Resources without blocks: 839
- Blocks without vectors: 86

### Key Files
- `N5/cognition/vectors_v2.db` — Primary vector store
- `N5/cognition/brain.db` — Legacy vector store + graph (entities, relationships)

## Tasks

### 1. BACKUP first (P35: Version, Don't Overwrite)
```bash
cp /home/workspace/N5/cognition/vectors_v2.db /home/workspace/N5/cognition/vectors_v2.db.bak.$(date +%Y%m%d)
cp /home/workspace/N5/cognition/brain.db /home/workspace/N5/cognition/brain.db.bak.$(date +%Y%m%d)
```

### 2. Write and run cleanup script for vectors_v2.db

The script must:

a. **Identify stale resources**: Query all resources, check `os.path.exists(path)`, collect IDs of stale ones.

b. **Delete stale resources cascade**: For each stale resource_id:
   - Delete from `vectors` where `block_id IN (SELECT id FROM blocks WHERE resource_id = ?)`
   - Delete from `blocks` where `resource_id = ?`
   - Delete from `tags` where `resource_id = ?`
   - Delete from `resources` where `id = ?`

c. **Delete orphaned resources** (resources with zero blocks):
   ```sql
   DELETE FROM resources WHERE id NOT IN (SELECT DISTINCT resource_id FROM blocks);
   ```

d. **VACUUM** the database after cleanup:
   ```sql
   VACUUM;
   ```

e. **Log counts** before and after for each table.

### 3. Write and run cleanup script for brain.db

Same pattern as above, plus:
- Delete orphaned blocks (blocks with no corresponding vector):
  ```sql
  DELETE FROM blocks WHERE id NOT IN (SELECT block_id FROM vectors);
  ```
- **DO NOT** touch the `entities` or `relationships` tables — the graph store is separate and healthy.

### 4. Verify integrity post-cleanup

Run integrity checks:
```sql
-- vectors_v2.db
SELECT COUNT(*) FROM resources;
SELECT COUNT(*) FROM blocks;
SELECT COUNT(*) FROM vectors;
SELECT COUNT(*) FROM resources WHERE id NOT IN (SELECT DISTINCT resource_id FROM blocks);  -- should be 0
SELECT COUNT(*) FROM blocks WHERE id NOT IN (SELECT block_id FROM vectors);  -- should be 0

-- brain.db (same checks)
```

### 5. Report metrics

## Important Constraints
- **NEVER delete entities or relationships from brain.db** — those are the graph store and are healthy
- **ALWAYS backup before any destructive operation**
- **Use transactions** — wrap the entire cleanup in a BEGIN/COMMIT
- **Log everything** — print counts at each step

- [ ] Orphaned blocks purged from brain.db
