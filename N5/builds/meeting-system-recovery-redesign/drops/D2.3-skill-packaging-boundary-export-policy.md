# Drop Brief: Skill Packaging Boundary + Export Policy

**Mission:** Define how the reusable meeting-system capability becomes a shareable skill inside this build while keeping private operating data out of the skill via an allowlist export model, simple file-based markers, and a bootstrap/setup contract for new workspaces.

**Output:** `N5/builds/meeting-system-recovery-redesign/artifacts/skill-packaging-and-export-policy.md`

---

## Context

V wants the reusable system capability created as a skill in this same build, not deferred. The skill should package the durable machinery of the meeting system — intake contracts, normalization logic, routing rules, block-selection logic, sqlite rebuild/sync logic, knowledge handoff logic, docs, prompts, and export tooling — while keeping real meeting data and private operating state outside the exported skill.

The packaging model should follow these principles:
- **system capability becomes the skill**
- **workspace state remains private operating data**
- export should use an **allowlist** model, not a broad export with exclusions
- file-based sentinel conventions are acceptable and should align with existing marker usage where sensible
- there should be a bootstrap/setup layer for new Zo/workspace installs because some required system paths/configuration will live outside the skill itself

This drop is a design/spec drop, not an implementation drop. It should define what the execution phase will create.

## Technical Concepts

| Concept | V's Level | Why It Matters Here |
|---------|-----------|---------------------|
| Skill boundary | Intermediate | Reusable logic must be separable from private state to be shareable safely |
| Allowlist export | Intermediate | Safer to explicitly export only approved files than to try to redact everything later |
| Bootstrap/setup contract | Intermediate | A skill alone may not be enough; a new workspace needs deterministic setup of external state paths and markers |

## Requirements

- Define the recommended skill slug and note if it is still a final-approval choice
- Specify the boundary between reusable capability and private operating data
- Include an allowlist export model with default-deny behavior for uncertain files
- Recommend simple file markers aligned with existing marker usage, including how `.n5protected` relates to export and what additional local markers are needed
- Specify expected private directories/state that must never export (real transcripts, DBs, logs, caches, raw pulls, manifests with names)
- Specify what sanitized fixtures/examples are allowed inside the skill
- Specify bootstrap/setup responsibilities for new environments, including any external workspace paths or config scaffolding required
- Note what export tooling should exist during execution (for example an `export_skill.py --dry-run` flow)

## Success Criteria

- [ ] Skill boundary is explicit

## Anti-Patterns (Avoid)

- Don’t package real transcripts or manifests into the skill
- Don’t rely on denylist-only export logic
- Don’t assume the skill can function in a new workspace without bootstrap/setup notes
- Don’t invent marker conventions without referencing existing system patterns where applicable
