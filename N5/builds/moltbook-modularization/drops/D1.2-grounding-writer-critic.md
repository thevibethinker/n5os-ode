# Drop Brief: Grounding + writer + harsh critic extraction

**Mission:** Extract writer-generation logic out of `direct_poster.py`, add real grounding from persona/context/concept history, and implement a harsh/pruning critic that aggressively rejects weak or repetitive drafts.

**Output:** Modular writer/grounding/critic code plus the corresponding `direct_poster.py` integration points.

---

## Context

The current system mostly writes from an embedded persona brief and ad hoc prompt scaffolding. V wants real persona leverage, semantic/context grounding, and stronger use of stored concepts. The quality bar should become much harsher: if a draft is weak, repetitive, vague, or conceptually stale, the system should kill it rather than post it.

## MUST DO

- Create `grounding.py`, `writer.py`, and `critic.py` under `Skills/zode-moltbook/scripts/posting_pipeline/`
- Move prompt construction and model-call helpers out of `direct_poster.py`
- Load actual persona/context inputs where feasible, with graceful fallbacks if dynamic grounding fails
- Reuse recent post concepts/history as part of writer grounding or critic duplicate checks
- Implement harsh/pruning draft rejection with explicit reasons
- Preserve comment generation path as part of the new writer boundary if practical

## MUST NOT DO

- Do not own feed-sensing and opportunity-selection logic beyond consuming their outputs
- Do not re-architect staging/publish mechanics
- Do not weaken gates to preserve volume; pruning is preferred in this build
- Do not touch feedback-loop ownership beyond what grounding needs as read-only input

  Given: the old writer leaned on a hardcoded `PERSONA_BRIEF`
  When: the new writer path is used
  Then: it uses richer grounding inputs from files/context/history, with explicit fallback behavior
  Verify: inspect `grounding.py` and `writer.py`, then run `python3 Skills/zode-moltbook/scripts/direct_poster.py run --dry-run`

S2: Harsh critic prunes weak drafts
  Given: V asked for a harsh/pruning approach

  Verify: inspect `critic.py` and demonstrate rejection behavior through dry-run logs or a targeted smoke check

  Given: comments share much of the same grounding/writer infrastructure
  When: the comment path executes

  Verify: dry-run path or isolated invocation shows comment generation still resolves through the new writer boundary
