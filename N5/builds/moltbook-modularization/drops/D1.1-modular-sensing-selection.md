# Drop Brief: Modular sensing + selection extraction

**Mission:** Extract sensing and selection concerns out of `direct_poster.py`, create a modular pipeline boundary for multi-surface feed sensing and smarter opportunity/submolt selection, and keep the legacy entrypoint thin.

**Output:** Updated modular code under `Skills/zode-moltbook/scripts/posting_pipeline/` plus a thinner `Skills/zode-moltbook/scripts/direct_poster.py`.

---

## Context

The current system over-relies on a thin feed snapshot and simplistic selection/routing logic. V specifically called out the weakness of reading only a small number of recent posts and questioned whether current thread selection is meaningful. This drop owns the top-of-funnel quality problem: what the system reads, how it summarizes it, and how it decides what threads or submolts are worth pulling on.

## MUST DO

- Create `sensing.py` and `selection.py` under `Skills/zode-moltbook/scripts/posting_pipeline/`
- Move feed reading / discourse context / submolt candidate extraction logic out of `direct_poster.py`
- Implement broader sensing across multiple feed surfaces and/or deeper windows using existing Moltbook reader capabilities
- Add a clearer opportunity-selection path for post topics and comment targets
- Keep `direct_poster.py` as the stable orchestrator entrypoint
- Preserve existing dry-run usability

## MUST NOT DO

- Do not own persona grounding, semantic memory, or writer prompt architecture beyond the interfaces needed for handoff
- Do not own harsh/pruning critic logic beyond integration hooks
- Do not change publish/staging semantics except where imports/interfaces require it
- Do not touch unrelated Moltbook files outside your declared scope unless strictly needed for interface compatibility
