# Drop Brief: Feedback loop + concept/performance reuse

**Mission:** Feed concept history and performance signals back into future generation and selection decisions so the system reuses what it has learned without collapsing into repetition.

**Output:** Feedback module integration plus additive metadata/logging if needed.

---

## Context

V explicitly asked whether stored concepts are cycled back into the system. Right now concept extraction exists, but it is only weakly present in the upstream decision path. This drop makes concepts and outcomes first-class inputs to future behavior while avoiding naive “repeat what worked last time” bias.

## MUST DO

- Create `feedback.py` under `Skills/zode-moltbook/scripts/posting_pipeline/`
- Read recent concept history and outcome signals from the social DB / existing logs
- Produce reusable summaries/signals for writer and selector consumers
- Bias the system away from exhausted lenses/concepts and toward underused high-signal territory
- Preserve analytics continuity and additive metadata design

## MUST NOT DO

- Do not own initial sensing/selection extraction or writer/critic extraction except through integration hooks
- Do not change publish mechanics unless required for additive metadata plumb-through
- Do not let short-term metrics become the sole decision source

## scope.files
- `Skills/zode-moltbook/scripts/posting_pipeline/feedback.py`
- `Skills/zode-moltbook/scripts/direct_poster.py`
- `Skills/zode-moltbook/scripts/staging_queue.py`
- `Skills/zode-moltbook/scripts/moltbook_poster.py`

## scope.responsibilities
- Concept-history reuse
- Performance-summary reuse
- Additive metadata handoff where required
- Integration of feedback outputs into orchestrator/module calls

## scope.must_not_touch
- Primary ownership of sensing/selection module internals
- Primary ownership of grounding/writer/critic module internals

## Scenarios

S1: Concepts become upstream inputs
  Given: concept extraction already exists downstream
  When: the feedback module runs
  Then: concept history is summarized and made available before generation, not only after publish
  Verify: inspect `feedback.py` and dry-run logs for concept-history summaries

S2: Outcome signals are additive, not dominant
  Given: recent performance matters but can create myopic loops
  When: performance summaries are produced
  Then: they influence but do not fully determine future selection/writing
  Verify: summary outputs contain balanced inputs (concept exhaustion + performance + recency) rather than a single score

S3: Analytics continuity is preserved
  Given: historical comparisons depend on existing logs/DB paths
  When: feedback metadata is added
  Then: existing publish logging still works and new metadata is additive rather than destructive
  Verify: inspect touched files and run dry-run integration without breaking existing log writes

## Expected Output

- Feedback module that surfaces reusable concept/performance context
- Additive integration into the orchestrator path
- Deposit summarizing feedback signals, metadata changes, and anti-bias protections
