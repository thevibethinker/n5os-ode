# D2.2: Contradiction Detector CLI

## Objective
Create a CLI tool that detects when new learnings contradict existing ones, flagging conflicts for human resolution rather than silent overwrite.

## Context
Without contradiction detection, newer learnings silently replace older ones even when they conflict. This can propagate errors. We need to catch conflicts like:
- Learning A: "API requires auth for all endpoints"
- Learning B: "Public endpoints work without auth"

## Requirements

### 1. Create Contradiction Detector Script
Location: `N5/scripts/contradiction_detector.py`

### 2. Detection Approach

**Semantic Similarity Check:**
Use sentence embeddings to find learnings that are topically related (similarity > 0.7).

**Contradiction Signals:**
For related learnings, check for contradiction indicators:
- Negation patterns: "does/does not", "can/cannot", "is/is not"
- Opposite values: "required/optional", "always/never", "must/should not"
- Conflicting numbers: "takes 5 minutes" vs "takes 2 hours"

**LLM Verification (for high-similarity pairs):**
Use `/zo/ask` to verify: "Do these two statements contradict each other? Statement A: ... Statement B: ..."

### 3. CLI Interface
```bash
# Check if new learning contradicts existing
python3 contradiction_detector.py check "new learning text"

# Scan all learnings for internal contradictions
python3 contradiction_detector.py scan

# Scan specific build learnings
python3 contradiction_detector.py scan --build <slug>

# Compare two specific texts
python3 contradiction_detector.py compare "text A" "text B"
```

### 4. Output Format
```
CONTRADICTION CHECK
===================

New Learning: "The API allows unauthenticated access to public endpoints"

⚠️ POTENTIAL CONTRADICTION FOUND

Existing Learning (index 3, added 2026-01-20):
  "All API endpoints require authentication tokens"
  
Similarity: 0.85
Contradiction Confidence: HIGH

Signals:
  - Negation: "allows unauthenticated" vs "require authentication"
  - Topic overlap: Both discuss API authentication

Recommendation:
  1. Verify which learning is correct
  2. Use `pulse_learnings.py invalidate` on the incorrect one
  3. Or use `pulse_learnings.py dispute` if context-dependent
```
